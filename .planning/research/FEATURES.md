# Feature Landscape: v1.8 Output Taxonomy & Cluster Quality

**Domain:** Obsidian vault export taxonomy for graphify  
**Researched:** 2026-04-28  
**Scope:** v1.8 only. Assumes v1.7 shipped profile composition, profile-driven routing, template conditionals/loops, dry-run/doctor, merge safety, orphan handling, diagram/vault promotion, and self-ingestion hardening.

## Summary

v1.8 should make the default vault output recognizable as graphify-owned, easier to navigate, and safer to migrate into the real `work-vault` to `ls-vault` workflow. The milestone should refine the existing `to_obsidian()` pipeline instead of creating a second writer: use profile defaults, mapping classification, templates, merge manifests, dry-run, and doctor/preflight as the extension points.

The biggest behavior change is taxonomy clarity. Generated default output should move into Graphify-owned folders, communities should become MOC-only output, god-node notes should split into code-derived CODE notes versus concept MOCs, and repo identity should be visible wherever exported notes would otherwise collide or become contextless.

Cluster quality should be expressed as a simple profile setting that controls MOC creation, not as a new clustering algorithm. Concept naming should improve names when LLM labels are available but remain deterministic, cached, and reproducible when they are not.

## Anti-Features

Features to explicitly avoid in v1.8.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| New Obsidian writer | Duplicates `to_obsidian()`, bypasses merge/orphan safety, and risks divergent behavior from shipped profile features. | Extend the existing profile, mapping, template, and merge pipeline. |
| New clustering algorithm | The milestone is a cluster-quality floor, not graph clustering research. Replacing Leiden/Louvain changes unrelated graph semantics. | Filter/reroute community output after clustering using `clustering.min_community_size`. |
| Deleting legacy community files automatically | Removing `_COMMUNITY_*` notes could destroy user edits or unmanaged notes. | Mark old managed outputs as orphans/conflicts through the existing merge manifest; document manual cleanup. |
| Uncached LLM-only community names | Names would churn, slow exports, and break wikilinks across reruns. | Use LLM names only when available and cache them behind deterministic fallback signatures. |
| Hidden repo identity inference only | Auto-derived names can be wrong in worktrees, renamed folders, or copied vaults. | Provide explicit CLI and profile controls with documented precedence. |
| Embedding repo identity everywhere | Pollutes concept MOC names and makes vault browsing noisy. | Apply repo identity primarily to CODE notes, tags, frontmatter, and manifests; concept MOCs stay human-readable. |
| One-shot destructive migration | Real vault migration needs reviewable steps, not a command that moves/deletes content. | Guide users through backup, validation, dry-run, export, review, and optional cleanup. |
| Expanding into Obsidian plugin behavior | v1.0 explicitly de-scoped `.obsidian/graph.json` and plugin-side management. | Keep output to standard Markdown, frontmatter, tags, Dataview, and manifests. |

## Feature Dependencies

```text
Graphify-owned layout -> MOC-only output -> migration guide examples
```

## Suggested Requirement Groups

### 1. Default Output Taxonomy

- Default profile MUST write generated notes under Graphify-owned folders.
- Default profile MUST separate CODE notes from concept MOCs.
- Existing custom profiles MUST continue to control folder placement through `folder_mapping` and mapping rules.
- Graphify MUST NOT manage `.obsidian/graph.json` as part of this milestone.

### 2. Community Output Semantics

- Community output MUST be MOC-only by default.
- Legacy community overview rendering MUST not be invoked by default.
- Existing legacy managed files MUST be handled by merge/orphan reporting, not deleted.
- Community frontmatter MUST retain useful fields already supported by templates, including `community`, `cohesion`, tags, and Dataview blocks.

### 3. Cluster-Quality Floor

- Profile schema MUST accept `clustering.min_community_size` as an integer greater than or equal to 1.
- Communities smaller than the threshold MUST be routed deterministically into a host MOC or Uncategorized MOC.
- Existing `mapping.moc_threshold` behavior SHOULD remain readable for compatibility, with `clustering.min_community_size` taking precedence when both are set.
- Isolates MUST NOT create standalone noisy MOCs unless the threshold explicitly allows size 1.

### 4. Concept Naming

- Concept MOC names MUST prefer supplied `community_labels`.
- When no supplied label exists, names MUST be deterministic from stable community content rather than raw community id alone.
- Naming fallbacks MUST use sanitized filenames and tags through existing safety helpers.
- Naming cache behavior MUST prevent unchanged communities from receiving different MOC filenames across reruns.

### 5. God-Node Taxonomy

- Code-derived god nodes MUST render as CODE notes, not generic Things or concept MOCs.
- Conceptual god nodes SHOULD contribute to concept MOCs rather than standalone CODE notes.
- CODE notes MUST link up to their parent concept MOC.
- Concept MOCs SHOULD link back to important CODE notes in that community.

### 6. Repo Identity

- Repo identity MUST resolve by precedence: CLI flag, profile key, auto-derived fallback.
- The resolved identity MUST be recorded in manifests and exported note frontmatter.
- CODE note filenames or tags SHOULD include repo identity by default to avoid collisions in multi-repo vaults.
- Concept MOC display names SHOULD remain repo-neutral unless a profile explicitly opts into repo-prefixed names.

### 7. Migration Guide

- Guide MUST cover the real `work-vault` to `ls-vault` flow step by step.
- Guide MUST start with backup and no-write validation.
- Guide MUST use `graphify --validate-profile`, doctor/preflight where available, and `--dry-run` before write commands.
- Guide MUST explain how to interpret CREATE, UPDATE, SKIP_PRESERVE, SKIP_CONFLICT, REPLACE, and ORPHAN outcomes.
- Guide MUST explain what changed from legacy `_COMMUNITY_*` output and how to safely clean up old managed notes.

## MVP Recommendation

Prioritize:

1. Graphify-owned default layout plus MOC-only community output.
2. `clustering.min_community_size` schema and deterministic tiny-cluster routing.
3. Repo identity precedence and CODE note propagation.
4. Hybrid concept naming with deterministic fallback.
5. Migration guide with dry-run-first real-vault steps.

Defer:

- Rich taxonomy dry-run summaries if core merge behavior needs more time; include a minimal summary in the migration guide instead.
- Advanced LLM naming UX such as manual approval queues; deterministic cached fallback is enough for v1.8.
- Any plugin-side Obsidian graph customization.

## Confidence

**Overall confidence:** High for feature boundaries and dependencies because the current code already exposes the right extension points in `profile.py`, `mapping.py`, `templates.py`, `merge.py`, and `export.py`.

**Needs phase-specific validation:** Exact folder names for the Graphify-owned layout, the final repo identity profile key name, and whether backward compatibility for `mapping.moc_threshold` is warning-only or fully documented in `validate_profile_preflight`.
- **Requiring mcp_excalidraw for any core functionality** — must remain optional
- **Overwriting user-edited diagram content** — when a `.excalidraw.md` already exists, graphify must check for user edits before overwriting (use file hash or timestamp)
