## Cross-Phase Integration Issues

### INT-01 (High): Approve flow bypasses vault manifest

**Location:** `graphify/__main__.py` lines 834-835
**Affected:** TRIP-01, TRIP-02, TRIP-03, TRIP-06, MCP-07

`_approve_and_write_proposal` calls `compute_merge_plan` and `apply_merge_plan` without passing `manifest=`, `force=`, `manifest_path=`, or `old_manifest=` parameters. This means:

1. **User-modified detection bypassed** — If a target note was edited by the user since the last `--obsidian` run, the approve flow will not detect this and may overwrite the note.
2. **Manifest not updated** — After writing the approved note, `vault-manifest.json` is not updated. The manifest drifts out of sync.
3. **SKIP_PRESERVE not triggered** — Modified notes that should receive SKIP_PRESERVE are treated as normal CREATE/UPDATE.
4. **Sentinel blocks in pre-existing files** — While `_synthesize_file_text` handles sentinels for UPDATE/REPLACE, the absent manifest means the flow never reaches the SKIP_PRESERVE path that would protect entire modified notes.

**Fix:** Load manifest before `compute_merge_plan`; pass `manifest_path` and `old_manifest` to `apply_merge_plan` in the approve wrapper, matching `export.py::to_obsidian` lines 600-612.

### INT-02 (Medium): Skill orchestrator missing auto_snapshot_and_delta

**Location:** `graphify/skill.md` Step 4 / Step 9
**Affected:** DELTA-01, DELTA-02

`auto_snapshot_and_delta` is exported from `snapshot.py`, registered in `__init__.py`, and wired to the `graphify snapshot` CLI subcommand. However, neither `skill.md` (the primary `/graphify` invocation path) nor the `--obsidian` CLI path calls it. Users get no automatic snapshot or delta report — they must manually run `graphify snapshot --delta`.

**Fix:** Add `auto_snapshot_and_delta(G, communities)` call after `to_json` in skill.md Step 4 or Step 9.

### INT-03 (Low): MCP get_node missing provenance fields

**Location:** `graphify/serve.py` lines 520-528
**Affected:** DELTA-04

`_tool_get_node` returns label, ID, source, type, community, and degree but not `extracted_at`, `source_hash`, or staleness classification. Agents cannot query a node's freshness through MCP.

**Fix:** Extend `_tool_get_node` response to include provenance fields; optionally expose `classify_staleness` as an MCP tool.

### INT-04 (Low): Agent-edges not queryable via MCP

**Location:** `graphify/serve.py`
**Affected:** MCP-08

`get_annotations` filters `annotations.jsonl` contents, but there is no MCP tool to list or filter `agent-edges.json` contents. Agents that previously called `add_edge` across sessions cannot retrieve those edges.

**Fix:** Add a `get_agent_edges` tool that returns `_agent_edges` with optional peer/session filters.
