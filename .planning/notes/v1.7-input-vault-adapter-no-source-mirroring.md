# v1.7 input: no source-path mirroring fallback

## The requirement

The configurable vault adapter must define an **explicit folder placement policy** for every note type (node, community overview, wiki article, MOC, etc.). It must **not** fall back to mirroring `source_file` paths under the output dir when the profile is silent on placement.

## Why this is non-negotiable

Source-path mirroring is what produced the v1.6-era self-ingestion bug (see `.planning/notes/obsidian-export-self-ingestion-loop.md`). When source paths drive output structure:

- Running from inside a vault produces nested `graphify-out/obsidian/graphify-out/obsidian/...` (self-ingest loop)
- Running against an external target vault produces folder structure that mirrors the *source codebase*, not the *vault's* organizational scheme — which defeats the entire purpose of having a vault adapter
- Round-trip safety becomes brittle: re-running graphify on a corpus whose source files moved would orphan or duplicate notes

## Acceptance criteria for v1.7 scoping

The v1.7 vault adapter contract must include:

- [ ] Profile schema requires a `placement:` block specifying destination folders per note type
- [ ] Default Ideaverse profile (used when no `.graphify/profile.yaml` is found) declares explicit Atlas/Statements/Sources/People folders — does not derive paths from `source_file`
- [ ] Validation step (`graphify --validate-profile`) errors if any note type lacks a placement rule
- [ ] `to_obsidian()` raises (or warns loudly + halts) rather than mirroring `source_file` if placement is unresolved
- [ ] Test: profile with missing placement for one note type → validation fails with actionable message
- [ ] Test: round-trip — moving a source file does not change destination note path (placement is profile-driven, not source-driven)
