# v1.8 Output Taxonomy & Cluster Quality — design reasoning

This note captures *why* the v1.8 decisions were made the way they were, so future planners can revisit assumptions if context shifts. Decisions themselves live in `MILESTONE-CONTEXT.md`; this is the trail of reasoning.

## Why deprecate `_COMMUNITY_*` notes entirely (option A over B)

**Surface evidence:** running graphify against a real Obsidian vault, the user couldn't tell which artifact to actually open — MOCs and `_COMMUNITY_*` overviews described the same cluster from different angles, gated by `mapping.moc_threshold: 3`.

**Code evidence:** `export.py:639-664` shows the per-community render path forks on `note_type` (`"moc"` or `"community"`), with both shapes living in the codebase indefinitely. v1.0 added MOCs as a *parallel* shape, never as a replacement — that intent was never finished.

**Tradeoff accepted:** option B (keep both, route to different folders) was rejected because it preserves the "two artifacts, same concept" confusion. Going to single-shape forces the cluster-quality work in F3/F4 to be load-bearing — every community is now a first-class MOC, so a singleton named "0" is unacceptable. That was a feature, not a bug, of choosing A: it forced the harder downstream work to happen in the same milestone.

## Why `min_community_size = 6` as default (not 10 as initially suggested)

User initially proposed 10. The shipped default is 6 because:

- 10 is aggressive on small vaults (<100 nodes) — most clusters would fall below floor and merge into `_Unclassified`, defeating the purpose
- 6 is the minimum count where Leiden's modularity becomes statistically meaningful for naming
- It's a profile key, so users can raise to 10+ on large codebase vaults without code change
- Empirically (v1.4 routing telemetry), median cluster size on graphify's own self-ingest is ~7

## Why hybrid (drop isolates, merge tiny → `_Unclassified`) over simpler options

Option 1 (drop everything sub-floor): clean but invisible — user loses the signal that those nodes exist as outliers.

Option 2 (single `_Unclassified` for everything sub-floor including isolates): keeps signal but treats degree-0 nodes (almost certainly noise) the same as 2-5 node connected clusters (might be real but undersized). Different signal-to-noise ratios deserve different handling.

Option 3 (re-cluster aggressively): blurs real distinctions to satisfy the floor, which is the wrong direction.

Option 4 (hybrid, the one chosen): respects that isolates and tiny-but-connected are different phenomena. Isolates are dropped; small connected clusters get an `_Unclassified` MOC where the user can still see them grouped, suggesting "these may need more context to cluster well."

## Why hybrid LLM naming with top-frequency-tag fallback (not god-node-label)

User initially gravitated toward "represent a concept visually" — that's a soft requirement that single-word god-node labels can't satisfy (`MOC: utils` doesn't represent any concept). Top-frequency tag is closer (`MOC: ml-models`) but depends on tag taxonomy being populated, which it isn't reliably across all extraction paths.

LLM naming is the only path that produces phrases like "Transformer Architecture" or "Authentication Pipeline" — concept phrases, not noun-stubs. The cost concern (10–30 calls per run) is bounded by community count, which is small.

**Determinism via cache:** SHA256 of sorted member-label set as cache key means a community with the same members re-runs to the same name. This matters because Obsidian wikilinks break if filenames drift across runs.

**Top-frequency-tag (not top-god-node) as fallback:** chosen because tags carry semantic intent the user controlled (via `tag_taxonomy` profile config), while top god-node label is a degree heuristic that often surfaces structural hubs (`utils`, `BaseClass`) over conceptual centers.

## Why two-class god-node taxonomy

The user's pain was visual: in a single Obsidian sidebar listing, you can't tell which notes came from tree-sitter AST extraction (deterministic, code-grounded) vs. LLM concept extraction (inferred, conceptual). Same icon, same folder, same naming convention — the vault becomes opaque.

**Why `CODE_` prefix specifically:** filename-level prefixes sort together in Obsidian's file explorer, giving instant visual grouping. Tags work too but require user to filter; prefix-grouping is passive.

**Why include `<repo-name>` in the prefix:** when a vault ingests multiple repos (the user's actual workflow), `CODE_AuthService.md` is ambiguous — which repo's AuthService? `CODE_portal-api_AuthService.md` is unambiguous and lets Obsidian's fuzzy-search scope by repo.

**Why bidirectional `concept_moc:` ↔ `members:` links:** this is what turns the graph into a navigable two-tier structure. From a CODE_ node you can click up to the concept MOC; from the MOC you can click down to specific implementations. Without bidirectionality, you can only navigate one direction (today: communities → nodes via MOC body wikilinks; nodes → community via community: integer field, but that integer is meaningless to the user).

## Why repo-name layered precedence (CLI > profile > auto-derive)

Three places wanting to set repo name:

- **Ad-hoc usage** (`graphify --repo-name foo` for a one-off run on someone else's repo): CLI flag is essential
- **Project-stable usage** (vault tied to one repo, never changes): profile key is essential
- **Zero-config first-run** (user just typed `graphify` in their repo dir): auto-derive from git remote or CWD basename is essential

Any one of these alone would frustrate at least one workflow. Layered precedence with CLI > profile > auto-derive is the standard cascade pattern (matches how every modern CLI handles this — git, gh, kubectl).

**Why git remote before CWD basename in auto-derive:** git remote names are stable across forks/clones (`origin` URL strips to canonical repo name), while CWD basename varies (`graphify-fork` vs `graphify-2` vs random renames). Git remote is the more authoritative signal when present.

## Open questions for plan-phase to resolve

These were flagged but not decided during exploration:

1. **MOC location**: `Atlas/Maps/` (Ideaverse default) or `Atlas/Sources/Graphify/MOCs/` (this milestone's proposal)? Trade Ideaverse compatibility for cleaner namespacing.
2. **F2 deprecation kind**: hard removal of `_COMMUNITY_*` code path, or sunset behind a `legacy_community_overview: true` flag for one milestone?
3. **CODE_ folder shape**: `Atlas/Sources/Graphify/Code/<repo>/` (nested) or `Atlas/Sources/Graphify/Code/` (flat)? Nested helps multi-repo, flat is simpler for single-repo users.
4. **Migration of existing vaults**: do we ship a one-shot migration command (`graphify migrate-output-layout`) to relocate existing notes, or rely on the merge engine's ORPHAN action and let users delete stale notes manually?

## What might invalidate these decisions

- If LLM naming proves prohibitively slow on large graphs (>100 communities), F4 may need to default to top-frequency-tag with LLM as opt-in.
- If users prefer the legacy `_COMMUNITY_*` shape (some may use it as raw exploratory material), F2 may need to be opt-out rather than removed entirely.
- If `git remote get-url origin` proves unreliable across CI/sandboxed environments, the auto-derive fallback may need to short-circuit faster to CWD basename.
