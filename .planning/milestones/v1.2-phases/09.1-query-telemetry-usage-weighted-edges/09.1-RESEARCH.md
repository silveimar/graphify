## User Constraints (from CONTEXT.md)

### Locked Decisions

**Telemetry Storage & Format**
- D-01: Aggregate edge counters in `graphify-out/telemetry.json` — a new sidecar file following the established pattern (alongside `annotations.jsonl` and `agent-edges.json`). No full query trace logs.
- D-02: Counter format is an edge→count map. Incremented on each BFS/DFS traversal in `serve.py`. Cleared on fresh graph build (new `graphify-out/` run).

**Edge Weight Mutation & Decay**
- D-03: Logarithmic scaling: `weight = 1.0 + log(traversals)`. Prevents runaway weights on popular edges while giving meaningful differentiation.
- D-04: Decay happens on graph rebuild — apply a decay multiplier (e.g., 0.8×) to all traversal counters when graphify re-runs. Aligns with the build cycle naturally.
- D-05: Floor 1.0 (no edge drops below structural weight), ceiling 10.0 (no single edge overwhelms graph algorithms). Decayed edges that hit counter=1 are effectively reset to baseline.

**Derived Edge Proposals**
- D-06: Derived shortcut edges auto-add to `agent-edges.json` with `confidence=INFERRED`. No human approval step. Safe because agent-edges.json is a sidecar — `graph.json` stays pristine.
- D-07: Threshold: 5 traversals of a 2-hop path (A→B→C) triggers a derived A→C edge. Configurable via telemetry.json metadata.
- D-08: 2-hop shortcuts only. No 3-hop or longer derived edges — diminishing signal quality at longer distances.

**Hot/Cold Surface in Reports**
- D-09: Hot paths and cold zones appear in `GRAPH_REPORT.md` only (new "Usage Patterns" section). Deterministic, no LLM needed. `GRAPH_ANALYSIS.md` stays focused on tournament insights.
- D-10: Percentile-based thresholds: hot = top 10% by traversal count, cold = bottom 10% (or zero traversals). Adapts to graph size automatically.

## Summary

Phase 9.1 adds usage-awareness to the graphify MCP server by recording which edges are traversed during BFS/DFS queries and persisting aggregate counters to a `telemetry.json` sidecar. The implementation has three functional layers: (1) telemetry recording in `serve.py`, (2) edge weight mutation applied at report generation time, and (3) reporting of hot/cold paths in `GRAPH_REPORT.md`.

The codebase already provides the exact patterns this phase needs. The sidecar read/write pattern (`_load_agent_edges` / `_save_agent_edges`) is the model for `telemetry.json`. The BFS/DFS functions already return edge tuples as `list[tuple[str, str]]` — telemetry hooks directly into these return values without modifying the traversal logic itself. The `report.generate()` function accepts the graph `G` plus arbitrary kwargs, so adding a `usage_data` optional parameter is the lowest-friction integration.

The key technical insight: all edge weights in the current codebase are `1.0` (set at extraction time in `extract.py`). This phase mutates these in the `generate()` call path — not in `graph.json` — following the established principle that `graph.json` is the read-only ground truth. The effective weight is computed on-the-fly from telemetry counters and applied transiently when rendering the report. The graph object `G` loaded from `graph.json` stays immutable.

**Primary recommendation:** Implement telemetry.json loading at `serve()` startup (alongside existing sidecar loads), instrument `_bfs()` and `_dfs()` return values in `_tool_query_graph()`, persist counters after each query, and pass usage data into `report.generate()` as a new optional parameter for the Usage Patterns section.

```python
def _save_agent_edges(out_dir: Path, edges: list[dict]) -> None:
    """Atomically write agent-edges.json to out_dir using os.replace."""
    out_dir.mkdir(parents=True, exist_ok=True)
    target = out_dir / "agent-edges.json"

        tmp.unlink(missing_ok=True)
        raise
```

### Pattern 3: Weight Formula Application

**What:** Logarithmic scaling `weight = max(1.0, min(10.0, 1.0 + log(traversals)))` applied when rendering the report — never mutating `graph.json`. [VERIFIED: formula from D-03; `log(1)=0`, so t=1 gives weight=1.0 exactly, matching the floor]

**Formula behavior verified:**
```python

# t=100 → weight=5.605
```

**Critical note:** `log(0)` is undefined — telemetry counters must never be 0 in the active counter set. Edges that decay to counter≤0 should be removed from the counter dict (or floor at 1 before formula application). This is the main edge case.

### Pattern 4: 2-Hop Derived Edge Detection

**What:** After each query recording, scan for A→B→C paths where counter["A:B"] >= threshold AND counter["B:C"] >= threshold. If A→C does not already exist in G or agent-edges, auto-add to `agent-edges.json` with `confidence=INFERRED`. [ASSUMED — implementation approach]

**When to use:** Post-recording, inside `_tool_query_graph()`.

```python
# Pattern [ASSUMED — not yet written]
def _check_derived_edges(
    G: nx.Graph,
    telemetry: dict,
    out_dir: Path,
    agent_edges: list[dict],
) -> None:
    """Propose derived 2-hop shortcut edges when traversal threshold met."""
    threshold = telemetry.get("threshold", 5)
    counters = telemetry.get("counters", {})
    existing = {(e["source"], e["target"]) for e in agent_edges}

    for key_ab, count_ab in counters.items():
        if count_ab < threshold:
            continue
        a, b = key_ab.split(":", 1)
        for neighbor in G.neighbors(b):
            key_bc = f"{b}:{neighbor}"
            if counters.get(key_bc, 0) >= threshold:
                pair = (a, neighbor)
                if pair not in existing and not G.has_edge(a, neighbor):
                    record = {
                        "source": a,
                        "target": neighbor,
                        "relation": "derived_shortcut",
                        "confidence": "INFERRED",
                        "confidence_score": 0.7,
                        "via": b,
                        "traversal_count": min(count_ab, counters[key_bc]),
                    }
                    agent_edges.append(record)
                    existing.add(pair)
    _save_agent_edges(out_dir, agent_edges)
```

### Pattern 5: Hot/Cold Percentile Calculation

**What:** Compute top/bottom 10% by traversal count for report display. [ASSUMED — implementation approach; `statistics.quantiles` is the standard stdlib tool]

```python
# In report.py [ASSUMED — not yet written]
import statistics

def _compute_hot_cold(
    G: nx.Graph,
    counters: dict[str, int],
    hot_percentile: float = 0.90,
    cold_percentile: float = 0.10,
) -> dict:
    """Return hot paths and cold edges for the Usage Patterns report section."""
    if not counters:
        return {"hot": [], "cold": [], "never": [], "total_queries": 0}

    all_counts = list(counters.values())
    total = sum(all_counts)

    if len(all_counts) >= 10:
        thresholds = statistics.quantiles(all_counts, n=10)
        hot_floor = thresholds[8]   # 90th percentile
        cold_ceil = thresholds[0]   # 10th percentile
    else:
        hot_floor = max(all_counts)
        cold_ceil = min(all_counts)

    hot = [(k, v) for k, v in counters.items() if v >= hot_floor]
    cold = [(k, v) for k, v in counters.items() if v <= cold_ceil and v > 0]
    never = [
        (u, v) for u, v, _ in G.edges(data=True)
        if f"{u}:{v}" not in counters and f"{v}:{u}" not in counters
    ]
    hot.sort(key=lambda x: -x[1])
    return {"hot": hot[:10], "cold": cold[:10], "never_traversed": len(never), "total_queries": total}
```

### Anti-Patterns to Avoid
- **Mutating `graph.json` with weight data:** The edge weights computed from telemetry are ephemeral display values, not permanent graph properties. Never write mutated weights back to `graph.json`. [VERIFIED: D-73 principle from STATE.md]
- **Recording traversals inside `_bfs()/_dfs()`:** Keep traversal functions pure. Recording belongs in `_tool_query_graph()` which owns the side-effectful server context.
- **Logging traversal per-edge in JSONL:** D-01 explicitly says no full query trace logs. Only aggregate counters in telemetry.json.
- **`log(0)` undefined:** Never pass counter=0 to the weight formula. Guard with `max(1, count)` before applying log, or strip zero-count entries during decay.
- **Edge key asymmetry:** The graph is undirected (`nx.Graph`, not `nx.DiGraph`). BFS/DFS will traverse edges in both (u,v) and (v,u) directions depending on traversal order. The edge key scheme must handle this — either normalize to `"min(u,v):max(u,v)"` or accept both directions as equivalent keys. [ASSUMED — needs decision: normalized vs. directional]

## Common Pitfalls

### Pitfall 1: `math.log(0)` crash on first traversal
**What goes wrong:** Counter starts at 0, first increment gives count=1, but if code applies log before increment (off-by-one), or if decay floors at 0 instead of removing the entry, `math.log(0)` raises `ValueError`.
**Why it happens:** Decay multiplier (e.g., 0.8 × 1 = 0.8, rounded down to 0 if using int) — must use float counters or strip sub-1 entries.
**How to avoid:** Keep counters as `int`, apply decay as `max(1, int(count * multiplier))` before deciding to keep/drop. Drop entries where `int(count * 0.8) == 0` (i.e., original count=1). Or: always apply `max(1, count)` before the log formula.
**Warning signs:** `ValueError: math domain error` in test suite or server logs.

### Pitfall 2: Undirected graph edge key collisions
**What goes wrong:** BFS returns edge (n2, n1) on one query and (n1, n2) on another. These get stored as separate counter keys, so the path appears to have half the traversals when computing hot paths.
**Why it happens:** `nx.Graph` (undirected) allows `G.neighbors(n2)` to return n1 and vice versa. `_bfs` appends `(n, neighbor)` tuples, so direction depends on traversal start.
**How to avoid:** Normalize edge keys: `key = f"{min(u,v)}:{max(u,v)}"`. Apply this in `_record_traversal` and in the hot/cold compute step.
**Warning signs:** Hot path counts are suspiciously halved; derived edges never trigger despite many queries.

### Pitfall 3: Derived edge duplication across server restarts
**What goes wrong:** Server restarts, loads agent-edges.json into memory, then immediately adds duplicate derived edges because threshold check doesn't consult the current agent-edges list.
**Why it happens:** `_check_derived_edges` builds `existing` set from in-memory `_agent_edges`, but if edges were added by a previous session they may or may not be in memory depending on reload behavior.
**How to avoid:** Build `existing` set fresh from the loaded `_agent_edges` list on every call (not cached). The `_load_agent_edges()` path already handles this.
**Warning signs:** agent-edges.json grows unboundedly with duplicate records.

### Pitfall 4: `generate()` signature change breaks skill.md callers
**What goes wrong:** `report.generate()` is called in at least three places in `skill.md` (lines 423, 484, 960 — verified by grep). If a new required parameter is added, all call sites in the skill.md must also be updated.
**Why it happens:** skill.md is a large file (50K lines) with multiple pipeline call sites.
**How to avoid:** Make `usage_data` an optional parameter with `None` default: `def generate(..., usage_data: dict | None = None)`. Skip the Usage Patterns section if `usage_data` is None. This preserves backward compatibility with existing call sites.
**Warning signs:** `TypeError: generate() missing required argument` in skill pipeline.

### Pitfall 5: Decay on rebuild clearing all counters
**What goes wrong:** Implementing decay as "delete telemetry.json on rebuild" instead of "multiply counters" — effectively resetting all history every run.
**Why it happens:** D-02 says "cleared on fresh graph build" which could be misread as "delete file". D-04 clarifies it means decay multiplier, not deletion.
**How to avoid:** Re-read D-04: decay multiplier, not wipe. The telemetry.json survives rebuilds but with reduced counter values. Only a full `graphify-out/` directory wipe clears telemetry completely.
**Warning signs:** Hot paths section always empty after re-run despite many queries.
