## Threat Verification

| Threat ID | Category | Disposition | Status | Evidence |
|-----------|----------|-------------|--------|----------|
| T-09.1-01 | Tampering | accept | CLOSED | Node ID pairs used as telemetry counter keys are derived from graph node IDs already sanitized by `security.py` at extraction time. No user-controlled input reaches edge keys. Accepted risk documented here. |
| T-09.1-02 | DoS | accept | CLOSED | Counter dict is bounded by graph edge count (finite). No per-query logging unbounded growth path. Accepted risk documented here. |
| T-09.1-03 | Tampering | mitigate | CLOSED | `graphify/serve.py:147` — `_check_derived_edges` uses `G.neighbors(b)` to enumerate only valid graph nodes as derived-edge targets. `serve.py:157` — `G.has_edge(a, neighbor)` blocks injection of edges that already exist. `serve.py:148` — `neighbor == a` guard eliminates self-loop proposals. Arbitrary edge injection is not possible. |
| T-09.1-04 | Information Disclosure | accept | CLOSED | `telemetry.json` is written to `graphify-out/`, the same directory as `graph.json`. Access level is identical. No new exposure surface. Accepted risk documented here. |
| T-09.1-05 | Injection | mitigate | CLOSED | `graphify/report.py:232,243` — all node labels in Usage Patterns table rows pass through `_sanitize_md()`. `report.py:249-253` — `_sanitize_md` replaces backticks with single quotes and HTML-escapes `<` and `>`. Markdown structure injection via label content is blocked. |
| T-09.1-06 | Tampering | accept | CLOSED | `graphify/report.py:65` — `usage_data` parameter is `dict | None = None`. Only consumed internally by the pipeline; no path from external input to this parameter. Accepted risk documented here. |
| T-09.1-07 | Tampering | accept | CLOSED | `graphify/skill.md:428-430,503-505,988-990` — all telemetry reads in skill.md are wrapped in `try/except (ValueError, OSError)` with fallback to None/default. Corrupt or tampered `telemetry.json` degrades gracefully without crashing the pipeline. `graphify-out/` is the pipeline's own output directory (same trust boundary as `graph.json`). Accepted risk documented here. |

## Accepted Risks Log

| Risk ID | Accepted Risk | Rationale |
|---------|--------------|-----------|
| T-09.1-01 | Telemetry counter keys derived from node IDs may reflect sensitive symbol names | Node IDs are already sanitized at extraction by `security.py`. Telemetry file is local-only in `graphify-out/`. No elevated exposure. |
| T-09.1-02 | Counter dict grows proportional to graph edge count | Bounded by graph size. No per-query unbounded accumulation path. Practical ceiling is the same as the graph edge set. |
| T-09.1-04 | `telemetry.json` exposes query traversal patterns | File lives in `graphify-out/`, identical access level to `graph.json` and `GRAPH_REPORT.md`. Not a new exposure boundary. |
| T-09.1-06 | `usage_data` parameter could be corrupted by caller | Parameter is optional with `None` default. All callers are internal pipeline stages; no external input path exists. |
| T-09.1-07 | `telemetry.json` in `graphify-out/` could be tampered between pipeline runs | Corrupt JSON handled by `try/except (ValueError, OSError)` in `skill.md`. Same trust boundary as all other `graphify-out/` artifacts. |

## Unregistered Threat Flags

None. All three SUMMARY files (`09.1-01-SUMMARY.md`, `09.1-02-SUMMARY.md`, `09.1-03-SUMMARY.md`) report no new threat surface beyond what was documented in the plans.

Notable deviation captured in `09.1-01-SUMMARY.md`: the `neighbor == a` self-loop guard added to `_check_derived_edges` during implementation was an auto-fix for a logic bug, not a new threat. The fix strengthens T-09.1-03 (already CLOSED).
