## Telemetry Storage & Format

| Option | Description | Selected |
|--------|-------------|----------|
| Aggregate counters | Simple edge→count map in a JSON file (like agent-edges.json pattern). Lightweight, easy to query for hot/cold. | ✓ |
| Full query traces | JSONL log of every query with full traversal path, timestamps, query text. Richer but grows unbounded. | |
| Both — traces + rollup | Log full traces to JSONL, periodically roll up into aggregate counters. Maximum flexibility but more complexity. | |

**User's choice:** Aggregate counters
**Notes:** Follows existing sidecar pattern. Sufficient for weight mutation and derived edge proposals.

| Option | Description | Selected |
|--------|-------------|----------|
| graphify-out/telemetry.json | New sidecar alongside annotations.jsonl and agent-edges.json. Cleared on fresh build. | ✓ |
| graphify-out/cache/telemetry.json | Under cache directory. Survives re-builds. | |
| In-memory only | Counters live only in running MCP server process. Lost on restart. | |

**User's choice:** graphify-out/telemetry.json
**Notes:** Consistent with established sidecar location pattern.

## Edge Weight Mutation & Decay

| Option | Description | Selected |
|--------|-------------|----------|
| Logarithmic scaling | weight = 1.0 + log(traversals). Diminishing returns, prevents runaway weights. | ✓ |
| Linear additive | weight = 1.0 + (traversals × increment). Simple but popular edges dominate. | |
| You decide | Claude picks formula during planning. | |

**User's choice:** Logarithmic scaling
**Notes:** None.

| Option | Description | Selected |
|--------|-------------|----------|
| On graph rebuild | Apply decay multiplier (e.g., 0.8×) when graphify re-runs. Natural cadence. | ✓ |
| Time-based (periodic) | Decay on a clock. Requires background process. | |
| Never decay | Counters only grow. Simplest but cold zones never emerge organically. | |

**User's choice:** On graph rebuild
**Notes:** Aligns with build cycle, no additional scheduling complexity.

| Option | Description | Selected |
|--------|-------------|----------|
| Floor 1.0, ceiling 10.0 | No edge below structural weight. Cap prevents algorithm domination. | ✓ |
| Floor 0.5, no ceiling | Allow decay below structural weight. More expressive but riskier. | |
| You decide | Claude picks bounds during implementation. | |

**User's choice:** Floor 1.0, ceiling 10.0
**Notes:** Decayed edges that hit 1.0 effectively reset to baseline.

## Derived Edge Proposals

| Option | Description | Selected |
|--------|-------------|----------|
| Auto-add to agent-edges.json | Derived edges go into existing sidecar with INFERRED confidence. No approval needed. | ✓ |
| Propose for approval | Staged for `graphify approve` before entering agent-edges.json. Safest but adds friction. | |
| Auto-add to graph.json | Direct into main graph. Most impactful but breaks read-only principle. | |

**User's choice:** Auto-add to agent-edges.json
**Notes:** Safe because agent-edges.json is a sidecar — graph.json stays pristine.

| Option | Description | Selected |
|--------|-------------|----------|
| 5 traversals | Moderate threshold. Low enough to be useful, high enough to avoid noise. | ✓ |
| 3 traversals | More aggressive — shortcuts emerge faster. Risk of incidental patterns. | |
| You decide | Claude picks threshold during implementation. | |

**User's choice:** 5 traversals
**Notes:** Configurable via telemetry.json metadata.

| Option | Description | Selected |
|--------|-------------|----------|
| 2-hop only | Only A→B→C → A→C shortcuts. Simplest, most meaningful. | ✓ |
| 2-hop and 3-hop | Also A→B→C→D shortcuts with higher threshold. More discovery but more noise. | |
| You decide | Claude determines during implementation. | |

**User's choice:** 2-hop only
**Notes:** Diminishing signal quality at longer distances.
