# Phase 9.1: Query Telemetry & Usage-Weighted Edges - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Track which MCP queries traverse which edges (query telemetry), maintain traversal counters per edge, and run a post-query pass that strengthens high-traffic paths and decays unused ones. After N traversals of A→B→C, propose a direct A→C derived edge with INFERRED confidence. Surface "hot paths" and "cold zones" in GRAPH_REPORT.md. Prerequisite for making multi-perspective analysis usage-aware.

</domain>

<decisions>
## Implementation Decisions

### Telemetry Storage & Format
- **D-01:** Aggregate edge counters in `graphify-out/telemetry.json` — a new sidecar file following the established pattern (alongside `annotations.jsonl` and `agent-edges.json`). No full query trace logs.
- **D-02:** Counter format is an edge→count map. Incremented on each BFS/DFS traversal in `serve.py`. Cleared on fresh graph build (new `graphify-out/` run).

### Edge Weight Mutation & Decay
- **D-03:** Logarithmic scaling: `weight = 1.0 + log(traversals)`. Prevents runaway weights on popular edges while giving meaningful differentiation.
- **D-04:** Decay happens on graph rebuild — apply a decay multiplier (e.g., 0.8×) to all traversal counters when graphify re-runs. Aligns with the build cycle naturally.
- **D-05:** Floor 1.0 (no edge drops below structural weight), ceiling 10.0 (no single edge overwhelms graph algorithms). Decayed edges that hit counter=1 are effectively reset to baseline.

### Derived Edge Proposals
- **D-06:** Derived shortcut edges auto-add to `agent-edges.json` with `confidence=INFERRED`. No human approval step. Safe because agent-edges.json is a sidecar — `graph.json` stays pristine (read-only pipeline ground truth per v1.1 principle).
- **D-07:** Threshold: 5 traversals of a 2-hop path (A→B→C) triggers a derived A→C edge. Configurable via telemetry.json metadata.
- **D-08:** 2-hop shortcuts only. No 3-hop or longer derived edges — diminishing signal quality at longer distances.

### Hot/Cold Surface in Reports
- **D-09:** Hot paths and cold zones appear in `GRAPH_REPORT.md` only (new "Usage Patterns" section). Deterministic, no LLM needed. `GRAPH_ANALYSIS.md` stays focused on tournament insights. Hot/cold data is available as tournament lens context if analysis runs separately.
- **D-10:** Percentile-based thresholds: hot = top 10% by traversal count, cold = bottom 10% (or zero traversals). Adapts to graph size automatically.

### Claude's Discretion
- Exact decay multiplier value (0.8× suggested, tune during implementation)
- Edge ID scheme for telemetry keys (e.g., `source:target` string key)
- Derived edge `relation` label (e.g., "derived_shortcut" or "usage_inferred")
- Derived edge `confidence_score` value (float for INFERRED edges)
- Report section formatting and ordering within GRAPH_REPORT.md
- Whether telemetry.json includes metadata (last_decay_timestamp, build_count, etc.)

</decisions>

### Established Patterns
- All edges have `weight: 1.0` today — no existing weight mutation anywhere in codebase
- Sidecar files use atomic write via `os.replace()` (see `_save_agent_edges`)
- Graph reload uses mtime check (`_reload_if_stale()`) — telemetry file can follow same pattern
- `graph.json` is never mutated by agents — sidecar pattern is sacred (v1.1 D-73)
