# Phase 8.1: Approve & Pipeline Wiring - Research

**Researched:** 2026-04-13
**Domain:** Python CLI mutation (approve subcommand), skill.md pipeline wiring
**Confidence:** HIGH

## Summary

Phase 8.1 closes two integration gaps left open after Phase 8: (1) the `graphify approve` path does not yet thread vault-manifest.json through the merge engine, so user-modified detection is bypassed on approval; and (2) `auto_snapshot_and_delta` exists in `snapshot.py` but is never called from `skill.md`, so snapshots and deltas do not generate automatically on pipeline runs.

All the machinery already exists. `merge.py::compute_merge_plan()` already accepts `manifest=` and `force=`. `merge.py::apply_merge_plan()` already accepts `manifest_path=` and `old_manifest=`. `merge.py::_load_manifest()` already handles missing/corrupt files gracefully. `snapshot.py::auto_snapshot_and_delta()` is complete and well-tested. The only work is (a) threading those parameters through the four indirection helpers in `__main__.py`, adding `--force` to the approve arg parser, deleting the proposal file on success (D-05), and (b) inserting two `auto_snapshot_and_delta` call sites into `skill.md`.

**Primary recommendation:** Do not refactor existing code. Wire existing APIs via additive changes only: update the three approve indirection helpers to accept/pass new kwargs, add manifest loading + `--force` flag to the approve subcommand, and insert the snapshot call after each `to_json` in skill.md.

### Locked Decisions

**D-01:** Manifest location is `graphify-out/vault-manifest.json` — same path used by `--obsidian` re-run.
**D-02:** `_approve_and_write_proposal()` must load vault-manifest.json before calling `_compute_merge_plan_for_approve()`, pass it as `manifest=` parameter, and pass `manifest_path=` + `old_manifest=` to `_apply_merge_plan_for_approve()`.
**D-03:** `graphify approve` supports a `--force` flag with identical semantics to `--obsidian --force`: bypasses user-modified detection (SKIP_PRESERVE), but sentinel blocks (`GRAPHIFY_USER_START/END`) are still preserved.
**D-04:** When approve encounters a user-modified target note (hash mismatch in manifest), apply `SKIP_PRESERVE` and print a warning to stderr: "Note X was user-modified, skipping. Use --force to override." The proposal stays pending so the user can re-run with `--force` or delete the target note.
**D-05:** After `graphify approve` successfully writes a note to the vault, the proposal file in `graphify-out/proposals/` is deleted. Proposals exist until approved or rejected (Phase 7 D-09), and successful write completes the lifecycle.
**D-06:** After successful write, vault-manifest.json is updated with the new content hash for the written note (atomic write via `os.replace()`).
**D-07:** `auto_snapshot_and_delta(G, communities, root)` is called in skill.md immediately after `to_json(G, communities, 'graphify-out/graph.json')` — around line 425 in the full pipeline path.
**D-08:** Snapshot/delta also runs in `--cluster-only` mode. A second `auto_snapshot_and_delta` call is added after the cluster-only `to_json` call (~line 947 in skill.md).
**D-09:** On first run (no prior snapshot exists), only the snapshot is saved. No delta is generated. `auto_snapshot_and_delta` already returns `None` for `delta_path` when no prior snapshot exists — the skill just skips delta reporting.
**D-10:** Skill output after snapshot/delta is a summary line: `"Snapshot saved (N nodes). Delta: +X added, -Y removed, Z migrated."` On first run: `"Snapshot saved (N nodes). No prior snapshot for delta comparison."` Keep it brief — user reads GRAPH_DELTA.md for details.

The three indirection helpers exist so tests can monkeypatch without touching import machinery. [VERIFIED: codebase read] Their current signatures:

### Pattern 2: Manifest Loading in `_approve_and_write_proposal`

Manifest path is `graphify-out/vault-manifest.json` (D-01). The function already receives `vault_path`; manifest lives alongside `graphify-out/` at a predictable location. [VERIFIED: Phase 8 D-08 in STATE.md: `manifest_path derived as vault_dir.parent/vault-manifest.json`]

The `--obsidian` CLI reads manifest as `vault_dir.parent / "vault-manifest.json"` where `vault_dir` is `graphify-out/obsidian`. For approve, the manifest path is simply `graphify-out/vault-manifest.json` (D-01 specifies same location regardless of `--vault` target).

A new indirection helper `_load_manifest_for_approve` should be added alongside the others for testability:

Note: `_load_manifest` is a module-private function in `merge.py`. Because tests monkeypatch the indirection helper, this is safe — the test never calls `_load_manifest` directly.

### Pattern 3: SKIP_PRESERVE Warning and Proposal Deletion

After calling `_apply_merge_plan_for_approve`, inspect the result for SKIP_PRESERVE actions and emit warnings. Per D-04, the proposal stays pending (not deleted) when skipped. Per D-05, deletion replaces the current "mark as approved + rewrite" pattern for successful writes.

The result from `apply_merge_plan` is a `MergeResult` dataclass with fields `succeeded`, `failed`, `skipped_identical`, and `plan`. The plan's `actions` list contains `MergeAction` objects with `action` field and `path`. [VERIFIED: codebase read of merge.py:1343-1357]

```python
result = _apply_merge_plan_for_approve(plan, vault_path, rendered, profile,
                                        manifest_path=manifest_path,
                                        old_manifest=old_manifest)

# Check for user-modified skips (D-04)
skipped_user_modified = [
    a for a in result.plan.actions
    if a.action == "SKIP_PRESERVE" and getattr(a, "user_modified", False)
]
if skipped_user_modified:
    for action in skipped_user_modified:
        print(
            f"[graphify] Note {action.path.name!r} was user-modified, skipping. "
            f"Use --force to override.",
            file=sys.stderr,
        )
    # Proposal stays pending — do NOT delete or mark approved
    return proposal

# On success: delete proposal file (D-05)
path.unlink()
return proposal
```

### Anti-Patterns to Avoid

- **Calling `merge.py::_load_manifest` directly from `__main__.py`:** The function is module-private (prefixed `_`). Always add an indirection helper `_load_manifest_for_approve` so tests can monkeypatch without touching internal import machinery. [VERIFIED: existing pattern at lines 775-792]
- **Rewriting `_apply_merge_plan_for_approve` with positional args:** Keep `manifest_path` and `old_manifest` as keyword-only (`*` separator) to match `apply_merge_plan`'s own signature.
- **Importing `auto_snapshot_and_delta` at module top in skill.md:** Skill.md Python blocks are executed via `-c` strings; all imports go inside the block to avoid context pollution.
- **Returning early on `SKIP_PRESERVE` without checking `user_modified`:** Not all SKIP_PRESERVE actions are user-modified (some may be SKIP_CONFLICT). Filter on `getattr(a, "user_modified", False)`.

## Common Pitfalls

### Pitfall 1: Proposal Stays as "approved" Instead of Deleted (D-05)

**What goes wrong:** The current `_approve_and_write_proposal` rewrites the proposal file with `status: "approved"`. D-05 requires deletion instead.
**Why it happens:** Phase 7 D-09 (proposal lifecycle) was written before D-05 (deletion on approval) was decided in Phase 8.1 discussion.
**How to avoid:** Replace the atomic rewrite block at `__main__.py:841-853` with `path.unlink()` for the success path.
**Warning sign:** Test `test_approve_and_write_proposal_calls_merge` verifies `status == "approved"` on the returned dict — this test needs to be updated to verify `path.exists() == False` instead of `on_disk["status"] == "approved"`.

### Pitfall 2: Manifest Path Derivation for Approve vs. Obsidian

**What goes wrong:** The `--obsidian` path derives manifest as `vault_dir.parent / "vault-manifest.json"` where `vault_dir` is `graphify-out/obsidian`. For approve, D-01 says always `graphify-out/vault-manifest.json` regardless of `--vault` target.
**Why it happens:** The `--vault` target is the user's Obsidian vault (not `graphify-out/obsidian`), so `vault_path.parent` would give the wrong directory.
**How to avoid:** Always construct manifest_path as `Path("graphify-out") / "vault-manifest.json"` inside `_approve_and_write_proposal`, not relative to `vault_path`.

### Pitfall 3: Missing `user_modified=True` Check on SKIP_PRESERVE Actions

**What goes wrong:** The merge engine may produce `SKIP_PRESERVE` for non-user-modified reasons (e.g., SKIP_CONFLICT with action="SKIP_PRESERVE"). Emitting the "user-modified" warning for all SKIP_PRESERVE would be misleading.
**Why it happens:** `MergeAction` uses a single `action` field; the `user_modified` attribute distinguishes the cause.
**How to avoid:** Filter with `getattr(a, "user_modified", False)` not just `a.action == "SKIP_PRESERVE"`.

### Pitfall 4: skill.md `auto_snapshot_and_delta` Root Path

**What goes wrong:** If `root=Path('.')` is used but the CWD during skill execution differs from the project root, snapshots land in the wrong directory.
**Why it happens:** The skill already uses `Path('graphify-out/...')` relative paths throughout. The snapshot function constructs `root / "graphify-out" / "snapshots"`.
**How to avoid:** Use `root=Path('.')` which matches the existing skill convention — the skill runs from the project root by design.

### Pitfall 5: Existing Test for Proposal Approval Breaks After D-05

**What goes wrong:** `test_approve_and_write_proposal_calls_merge` currently asserts `on_disk["status"] == "approved"` but after D-05 the file is deleted.
**Why it happens:** Test was written before D-05 was locked.
**How to avoid:** Update the test to check `not (proposals_dir / f"{record_id}.json").exists()` and verify the returned `proposal` dict (in-memory) has the correct data.
