## Phase Boundary

Thread vault manifest through `graphify approve` so approved proposals respect user-modified detection and manifest updates. <decisions>
## Implementation Decisions

### Manifest Threading for Approve
- **D-01:** Manifest location is `graphify-out/vault-manifest.json` — same location as `--obsidian` re-run (consistent with Phase 8 D-08). The approve flow reads from this path regardless of the `--vault` target path.
- **D-02:** `_approve_and_write_proposal()` must load vault-manifest.json before calling `_compute_merge_plan_for_approve()`, pass it as `manifest=` parameter, and pass `manifest_path=` + `old_manifest=` to `_apply_merge_plan_for_approve()`. This enables user-modified detection in the approve path.
- **D-03:** `graphify approve` supports a `--force` flag with identical semantics to `--obsidian --force`: bypasses user-modified detection (SKIP_PRESERVE), but sentinel blocks (`GRAPHIFY_USER_START/END`) are still preserved.

### Approve Conflict Behavior
- **D-04:** When approve encounters a user-modified target note (hash mismatch in manifest), apply `SKIP_PRESERVE` and print a warning to stderr: "Note X was user-modified, skipping. Use --force to override." The proposal stays pending so the user can re-run with `--force` or delete the target note.
- **D-05:** After `graphify approve` successfully writes a note to the vault, the proposal file in `graphify-out/proposals/` is deleted. Proposals exist until approved or rejected (Phase 7 D-09), and successful write completes the lifecycle.
- **D-06:** After successful write, vault-manifest.json is updated with the new content hash for the written note (atomic write via `os.replace()`).

### Snapshot & Delta Pipeline Wiring
- **D-07:** `auto_snapshot_and_delta(G, communities, root)` is called in skill.md immediately after `to_json(G, communities, 'graphify-out/graph.json')` — around line 425 in the full pipeline path. Graph and communities are finalized at this point.
- **D-08:** Snapshot/delta also runs in `--cluster-only` mode. A second `auto_snapshot_and_delta` call is added after the cluster-only `to_json` call (~line 947 in skill.md).
- **D-09:** On first run (no prior snapshot exists), only the snapshot is saved. No delta is generated. `auto_snapshot_and_delta` already returns `None` for `delta_path` when no prior snapshot exists — the skill just skips delta reporting.
- **D-10:** Skill output after snapshot/delta is a summary line: `"Snapshot saved (N nodes). Delta: +X added, -Y removed, Z migrated."` On first run: `"Snapshot saved (N nodes). No prior snapshot for delta comparison."` Keep it brief — user reads GRAPH_DELTA.md for details.

### Claude's Discretion
- Exact stderr warning message format for approve conflicts
- How `--force` is parsed in the approve subcommand (follows existing `sys.argv` pattern)
- Whether to print the proposal ID alongside the skip warning
- Python block formatting in skill.md for the auto_snapshot_and_delta call
- Snapshot cap parameter (default 10 from `snapshot.py`)

</decisions>

- `__main__.py` CLI uses `sys.argv` manual parsing — `--force` follows existing flag parsing pattern
