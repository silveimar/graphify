## Warnings

### WR-01: No lower bound on traversal depth allows negative values

**File:** `graphify/serve.py:535`
**Issue:** `depth = min(int(arguments.get("depth", 3)), 6)` enforces an upper bound of 6 but no lower bound. A caller passing `depth: -1` or `depth: 0` would result in no traversal beyond start nodes. While `depth: 0` is arguably valid (return only start nodes), negative values are nonsensical and should be clamped.
**Fix:**
```python
depth = max(1, min(int(arguments.get("depth", 3)), 6))
```

### WR-02: max_hops parameter not passed to nx.shortest_path as cutoff

**File:** `graphify/serve.py:628-634`
**Issue:** The `shortest_path` tool accepts a `max_hops` parameter but only checks the result length after computing the full shortest path. On large graphs, `nx.shortest_path` will explore the entire connected component even when the caller wants to limit search to 2-3 hops. More importantly, the API contract is misleading -- the parameter name suggests the search is bounded, but it is only a post-hoc filter. The `nx.shortest_path` function supports a `cutoff` parameter that would provide true bounded search.
**Fix:**
```python
try:
    path_nodes = nx.shortest_path(G, src_nid, tgt_nid, cutoff=max_hops)
except (nx.NetworkXNoPath, nx.NodeNotFound):
    return f"No path found between ..."
# The cutoff parameter already ensures len(path_nodes) - 1 <= max_hops,
# so the post-hoc check can be removed or kept as a safety net.
```

### WR-03: ISO-8601 timestamp comparison via string ordering is fragile

**File:** `graphify/serve.py:186-189`
**Issue:** `_filter_annotations` compares timestamps using string ordering (`>=`, `<=`). This works correctly only when all timestamps use the same timezone offset format. The server generates timestamps with `datetime.now(timezone.utc).isoformat()` which produces `+00:00` suffix, but user-provided `time_from` / `time_to` values could use `Z` suffix. Since `Z` (0x5A) > `+` (0x2B) in ASCII, `"2026-01-01T00:00:00Z" > "2026-01-01T00:00:00+00:00"` evaluates to `True`, which is semantically incorrect (they represent the same instant).
**Fix:**
```python
def _normalize_ts(ts: str) -> str:
    """Normalize Z suffix to +00:00 for consistent string comparison."""
    if ts.endswith("Z"):
        return ts[:-1] + "+00:00"
    return ts

def _filter_annotations(annotations, peer_id, session_id, time_from, time_to):
    result = list(annotations)
    if peer_id is not None:
        result = [r for r in result if r.get("peer_id") == peer_id]
    if session_id is not None:
        result = [r for r in result if r.get("session_id") == session_id]
    if time_from is not None:
        tf = _normalize_ts(time_from)
        result = [r for r in result if _normalize_ts(r.get("timestamp", "")) >= tf]
    if time_to is not None:
        tt = _normalize_ts(time_to)
        result = [r for r in result if _normalize_ts(r.get("timestamp", "")) <= tt]
    return result
```

**File:** `tests/test_serve.py:363`

```python
source = Path(serve_mod.__file__).read_text()
```
