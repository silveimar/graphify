# Phase 8.2: MCP Query Enhancements - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend MCP server so agents can query node staleness state (FRESH/STALE/GHOST) and retrieve agent-created edges. Two targeted enhancements to `serve.py`: (1) add provenance fields and live staleness classification to `get_node` response, (2) add a new `get_agent_edges` query tool with peer/session filtering. Closes INT-03 and INT-04 from the v1.1 milestone audit.

</domain>

<decisions>
## Implementation Decisions

### Provenance in get_node
- **D-01:** `get_node` response appends three new lines to the existing text format: `Extracted At: {iso_timestamp}`, `Source Hash: {sha256}`, `Staleness: {FRESH|STALE|GHOST}`. No breaking change to return type — agents already parse the multi-line text.
- **D-02:** Staleness is computed live by calling `classify_staleness()` from `graphify.delta` on the node's data dict. This requires a `Path.exists()` + optional `stat()` call — negligible overhead for a single-node query.
- **D-03:** If a node has no provenance metadata (e.g., concept nodes, LLM-extracted nodes without source files), `get_node` shows `Extracted At: —`, `Source Hash: —`, `Staleness: FRESH` (matches `classify_staleness()` default behavior for missing provenance).

### Agent-Edges Query Tool
- **D-04:** New MCP tool `get_agent_edges` registered in `_handlers` dict alongside existing tools. Returns agent-edges from the in-memory `_agent_edges` list (loaded at startup from `agent-edges.json`).
- **D-05:** Filter parameters mirror `get_annotations` pattern (Phase 7 D-06): optional `peer_id` and `session_id` string filters. Returns all agent-edges when no filters provided.
- **D-06:** Return format is JSON array of edge records (same shape as stored in `agent-edges.json`). Consistent with `get_annotations` returning JSON.
- **D-07:** No time range filter for agent-edges — they are fewer and more permanent than annotations. Keep the tool simple. If time filtering is needed later, it can be added without breaking the interface.

### Claude's Discretion
- Exact JSON Schema for `get_agent_edges` inputSchema (property names, required vs optional)
- Whether to add a `node_id` filter to `get_agent_edges` (filter edges involving a specific node) — optional enhancement if it simplifies common queries
- Tool description text for `get_agent_edges` in the `@server.list_tools()` registration
- Import style for `classify_staleness` (top-level vs lazy import inside handler)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `delta.py::classify_staleness(node_data)` — ready-to-use staleness classifier, just import and call with node data dict
- `serve.py::_filter_annotations()` — filter helper for annotations; agent-edges filter can follow same pattern
- `serve.py::_agent_edges` list — already loaded at startup, already updated by `add_edge` tool; query tool just reads it
- `serve.py::_reload_if_stale()` — mtime-based graph reload; `get_node` already calls this before reading

### Established Patterns
- Tool registration: `types.Tool(name=..., description=..., inputSchema={...})` in `@server.list_tools()`
- Tool handler: sync function in `_handlers` dict returning string (text or `json.dumps()`)
- Filter pattern: optional string params, iterate + match, return `json.dumps(results)`
- Graph node access: `G.nodes(data=True)` returns `(nid, data_dict)` pairs

### Integration Points
- **serve.py `_tool_get_node()`**: Add 3 lines after existing output (line 528). Import `classify_staleness` from `graphify.delta`.
- **serve.py `@server.list_tools()`**: Register `get_agent_edges` tool with inputSchema.
- **serve.py `_handlers` dict**: Add `"get_agent_edges": _tool_get_agent_edges` entry.

</code_context>
