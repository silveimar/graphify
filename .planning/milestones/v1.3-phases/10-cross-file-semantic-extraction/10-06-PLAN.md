    **A. Add `_load_dedup_report` helper** in `graphify/serve.py`, placed alongside the existing `_load_telemetry` / `_load_agent_edges` helpers (around line 81-100):

    ```python
    def _load_dedup_report(out_dir: Path) -> dict[str, str]:
        """D-16: load {eliminated_id: canonical_id} alias map from dedup_report.json.

        Returns {} if the report is missing, unreadable, malformed, or has no
        alias_map key. Never raises — broken dedup report must not crash MCP serve.
        """
        path = out_dir / "dedup_report.json"
        if not path.exists():
            return {}
        try:
            data = json.loads(path.read_text(encoding="utf-8"))
        except (json.JSONDecodeError, OSError):
            return {}
        alias_map = data.get("alias_map", {}) if isinstance(data, dict) else {}
        # Defensive: ensure all keys/values are strings (reject anything else)
        if not isinstance(alias_map, dict):
            return {}
        return {str(k): str(v) for k, v in alias_map.items() if isinstance(k, str) and isinstance(v, str)}
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dedup_report.json → serve.py | File is generated by graphify.dedup.write_dedup_reports. In principle an attacker who can write to graphify-out/ could craft a malicious alias_map — but that attacker already has code execution on the host. Defensive: reject non-string keys/values. |
| agent query → _run_query_graph | Agent (possibly untrusted / adversarial in multi-tenant MCP scenarios) can supply any node_id string. Alias lookup is a pure dict lookup — no code path for injection. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-06 | Information Disclosure | `merged_from` / `resolved_from_alias` leaking internal node IDs to MCP caller | accept | This is INTENTIONAL per D-16 — the agent MUST see the redirect to understand why a query for `auth` returned `authentication_service`. The `merged_from` list is a feature, not a leak. Test `test_run_query_graph_resolves_alias` confirms the field is present. No PII risk: node IDs are slugified from code entity names, not user data. |

Additional defense-in-depth:
- `_load_dedup_report` rejects non-string values in alias_map, preventing a malformed report from crashing the serve loop (availability property).
- All file reads go through try/except around JSON parsing; broken report → `{}` fallback → identical to pre-Phase-10 behavior.
