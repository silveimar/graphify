## Tool Contract Shape

### Q1: How should the Phase 9.2 retrieval tool be surfaced in the MCP schema?

| Option | Description | Selected |
|--------|-------------|----------|
| Extend query_graph | Add budget, layer, continuation_token to existing tool; old callers unchanged; hybrid text + JSON metadata response. | ✓ |
| New tool query_graph_progressive | Ship new tool alongside query_graph; freeze old; doubles surface area. | |
| Three separate tools (L1/L2/L3) | query_graph_summary / _edges / _subgraph; most explicit but forces client-side orchestration. | |

**User's choice:** Extend query_graph (Recommended)
**Notes:** Lowest-friction upgrade path; backward compatible with callers that still pass `token_budget=2000`. Unified tool keeps the agent's mental model simple and prevents MCP tool-surface explosion.

### Q2: How should agents request Layer 2/3 after receiving a Layer 1 response?

| Option | Description | Selected |
|--------|-------------|----------|
| Opaque continuation_token | base64(JSON) encoding prior query + frontier + graph mtime hash; stateless server. | ✓ |
| Explicit layer + node_ids parameters | Agent tracks frontier and re-passes on each call; transparent but burdens client. | |
| Server-side session_id | Simplest client, but server holds state; conflicts with mtime-reload pattern. | |

**User's choice:** Opaque continuation_token (Recommended)
**Notes:** Matches D-13 mtime-reload semantics from v1.1 — any stale token surfaces `status="graph_changed"` cleanly instead of serving inconsistent data.

## Cardinality Estimation Approach

### Q1: How should TOKEN-02 estimate result cardinality before a multi-hop query runs?

| Option | Description | Selected |
|--------|-------------|----------|
| Exact BFS probe at depth-1 | 1-hop frontier + branching-factor extrapolation; cached at load. Accurate ~20%, O(seed-neighborhood). | ✓ |
| Community-stats proxy | Sum seed-community sizes as upper bound; O(seeds) cost but loose; may trigger false aborts. | |
| Reservoir sampling | K random walks to depth N; tunable but adds knobs and marginal gains under 10k-node graphs. | |

**User's choice:** Exact BFS probe at depth-1 (Recommended)
**Notes:** Graph sizes we care about (500-5000 nodes) make even the "expensive" option cheap in absolute terms. Accuracy matters more than savings here since false-positive aborts erode agent trust.

## Bidirectional BFS Trigger & Edge Cases

### Q1: When should bidirectional BFS activate per TOKEN-03?

| Option | Description | Selected |
|--------|-------------|----------|
| Auto at depth >=3 | Matches requirement literally; no flag, no heuristic; predictable telemetry. | ✓ |
| Auto at depth >=3 only when target is set | Cleaner semantics but misses perf win on open-ended queries. | |
| Opt-in via strategy param | Max control but violates "agents shouldn't know internals". | |

**User's choice:** Auto at depth >=3 (Recommended)

### Q2: What should happen when the two BFS frontiers never meet?

| Option | Description | Selected |
|--------|-------------|----------|
| Return partial with status=incomplete | Traverse both sides up to budget; return partial + metadata (status, reason, frontier_nodes). | ✓ |
| Fall back to unidirectional from source | Auto-retry; higher hit rate but silently changes search_strategy mid-query. | |
| Return empty with explicit error | Cleanest contract but wastes partial traversal work. | |

**User's choice:** Return partial with status=incomplete (Recommended)
**Notes:** Keeps telemetry honest — one query, one search_strategy. Agent gets enough information (frontier node IDs) to decide whether retrying with larger budget is worth it.
