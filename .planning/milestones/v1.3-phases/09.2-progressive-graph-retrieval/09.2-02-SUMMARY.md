## Decisions Made

- **Hand-rolled bi-BFS vs wrapping `networkx._bidirectional_pred_succ`** — hand-rolled. The NetworkX helper is a private API (Pitfall 1), returns pred/succ dicts for path reconstruction rather than a visited set, and cannot express partial results on disjoint frontiers. CONTEXT D-07 explicitly forbids silent fallback, which means a 3-state status return is non-optional — matches only a hand-rolled implementation.
- **Duplicate edges at meet point** — not deduplicated inside `_bidirectional_bfs`. The raw list is returned so the caller can dedupe uniformly (Pitfall 6); Plan 03 will normalize to `{(min(u,v), max(u,v))}` before calling `_record_traversal`. Test `test_bidirectional_no_double_counting_at_meet` confirms dedup is achievable on the output.
- **K heuristic literal constant locked in code and test** — `max(3, min(20, int(math.log2(N))))`. For N=11 (the star-graph fixture) this gives K=3, and the test hard-asserts `len(out) == 3`. Future tuning requires a deliberate test update, which is the right ergonomic — calibration changes are plan-level decisions.
- **Determinism tie-break on target synthesis** — sorted by `(-degree, node_id)`. Matters because continuation tokens (Plan 01) hash query params; a non-deterministic target set would invalidate drill-down tokens across runs.
- **`_record_traversal` default kwarg is `"bfs"`** — matches pre-9.2 implicit behavior. The single existing call site at `serve.py:769` passes positional args only (`_record_traversal(_telemetry, edges)`), so no call-site change needed; backward compat is automatic.

## Known Nuances

- **AC6 literal grep is more permissive than intent** — the plan asked for `grep -q "_bidirectional_pred_succ" graphify/serve.py` to exit non-zero. The string appears in `_bidirectional_bfs`'s docstring explaining why we don't import the NetworkX private helper. A stricter `grep -E "^(import|from).*_bidirectional_pred_succ"` returns no matches, which is the intended contract (no actual import of private API). The docstring mention is intentional architectural reasoning and should stay — future maintainers searching "why don't we use NetworkX's built-in" will land in the right place.
- **Budget-exhausted overshoot behavior** — because the cap check fires at loop-top and one frontier expansion is fully executed before the next top-check, `len(visited)` can briefly exceed `max_visited` by roughly one frontier's worth. Test assertion is `len(visited) >= 5` rather than `== 5` to accommodate this. Documented in the function docstring and RESEARCH §Pattern 2.
