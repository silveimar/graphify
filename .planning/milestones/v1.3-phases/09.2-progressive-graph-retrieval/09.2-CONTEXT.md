# Phase 9.2: Progressive Graph Retrieval - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Token-aware progressive graph retrieval via MCP so agents can query 500+ node graphs without context blowout. Delivers three orthogonal mechanisms on the existing `query_graph` tool in `graphify/serve.py`:

1. **3-layer progressive response protocol** (TOKEN-01) — Layer 1 compact summaries (~50 tok/node), Layer 2 edge details + neighbors on drill-down (~200 tok/node), Layer 3 full subgraph with all attributes only on explicit request
2. **Cardinality estimation** (TOKEN-02) — pre-flight estimate (node count, edge count, approximate tokens) returned before multi-hop executes, with agent abort option
3. **Bidirectional BFS** (TOKEN-03) — depth ≥ 3 queries meet-in-the-middle from both endpoints, eliminating exponential path explosion

TOKEN-04 (Bloom filter skip + materialized 2–3-hop closure cache) is **out of scope** for Phase 9.2 and deferred to a follow-up phase pending UAT signal.

Out of scope: new extraction behavior, entity dedup (that's Phase 10), slash commands (that's Phase 11), anything that mutates `graph.json` (immutable per v1.1 contract).

</domain>

<decisions>
## Implementation Decisions

### Tool contract shape

- **D-01: Extend the existing `query_graph` MCP tool in place.** Add three new parameters: `budget: int` (total token ceiling for the response), `layer: int` (1|2|3, default 1), and `continuation_token: str` (opaque base64 JSON, optional). Existing callers that pass `token_budget=2000` keep working without behavior change. Old `token_budget` parameter is preserved as a deprecated alias for `budget`; behavior is unified.
- **D-02: Hybrid response format.** Layer N response returns the traditional text summary (current `_subgraph_to_text()` output, extended for layer-specific density) followed by a structured JSON metadata block containing `cardinality_estimate`, `search_strategy`, `status`, `continuation_token`, and `layer`. Keeps backward compatibility with current text-only consumers while giving new consumers machine-readable structure. Parseable by a simple split on a sentinel header.
- **D-03: Stateless drill-down via opaque `continuation_token`.** Layer 1 response includes a base64-encoded JSON token containing original query params, visited frontier, layer progression, and a content-hash of the graph mtime. Agent passes it back on the next call. Server validates the content-hash against current `_reload_if_stale()` mtime; mismatch returns a clean "graph_changed" error so the agent knows to re-query from scratch. No server-side session state.

### Cardinality estimation

- **D-04: Exact 1-hop BFS probe with branching-factor extrapolation.** Before any depth ≥ 2 query runs, do a 1-hop BFS from the seed nodes and extrapolate frontier size to requested depth using a precomputed graph-average-branching-factor cached at load time in `_load_graph()`. Cost is O(seed_neighborhood_edges) — milliseconds on 5k-node graphs. Token estimate multiplies node count by per-layer density constants (L1=50, L2=200, L3=full-attr-sum).
- **D-05: Cardinality always returned, abort is advisory.** The `cardinality_estimate` block returns on every multi-hop call regardless of whether the query executes. The agent decides based on its own budget math whether to proceed; the server never refuses on estimate alone (avoids false-positive rejections on under-estimated cheap queries). Exception: if estimate > 10x the supplied `budget`, server returns the estimate + empty result + `status="estimate_exceeded"` without running the full traversal.

### Bidirectional BFS

- **D-06: Auto-activate bidirectional BFS at depth ≥ 3 unconditionally.** No flag, no heuristic. Matches requirement TOKEN-03 literally and keeps telemetry predictable. When no explicit target is supplied, the "target set" is synthesized from remaining high-degree frontier nodes (top-K by degree, K proportional to graph size). New function `_bidirectional_bfs()` added alongside existing `_bfs()` (line 350) — old function stays as the single-endpoint path for depth < 3.
- **D-07: Disjoint-frontier handling returns partial with structured status.** When the two frontiers don't meet within the budget/depth cap, return whatever was traversed from both sides with `metadata.status="incomplete"` and `metadata.reason="frontiers_disjoint"` (or `"budget_exhausted"`). Include the explored frontier node IDs so the agent can decide whether to retry with larger budget. No auto-fallback to unidirectional — that would silently change `search_strategy` mid-query and muddy telemetry.
- **D-08: `search_strategy` field plugs into existing `_record_traversal()` telemetry.** Added as a new field on each traversal record at `serve.py:102`. Values: `"bfs"`, `"dfs"`, `"bidirectional"`. Surfaces in the "Usage Patterns" section of `GRAPH_REPORT.md` added in Phase 9.1 — no new reporting surface, unified observability.

### TOKEN-04 stretch — deferred

- **D-09: TOKEN-04 (Bloom filter + materialized 2–3-hop transitive closure cache) fully deferred to a follow-up phase.** Not in Phase 9.2 scope. Rationale: core protocol (TOKEN-01/02/03) alone is a large delivery; stretch was pre-tagged conditional; `_check_derived_edges()` at `serve.py:129` already provides scaffolding for the closure cache when the follow-up phase lands. Roadmap should add a placeholder follow-up phase (likely v1.3 Phase 9.3 or v1.4) driven by actual UAT complaints about deep-join cost.

### Claude's Discretion

- Exact continuation_token schema (fields, encoding, hash algorithm) — choose during planning, document in PLAN.md
- Per-layer token-density constants (the 50 / 200 / full-attr numbers) — calibrate against real graphs during planning and lock in tests
- Whether to store branching-factor at graph-load time as a module-level cache or on the `G` object — implementation detail
- Exact top-K for synthesized target set when no explicit target supplied in bi-BFS — calibrate against graph density
- Error message wording and log format for `status="graph_changed"`
- Whether to deprecate the `token_budget` alias immediately, on next milestone, or keep indefinitely — D-01 says unified for now; planner decides deprecation timing

</decisions>

## Existing Code Insights

### Reusable Assets
- **`query_graph` MCP tool (`serve.py:463`)** — existing contract; extended in-place per D-01. Already has `token_budget`, `depth`, `mode` params — minimal schema churn.
- **`_subgraph_to_text()` (`serve.py:382`)** — current post-traversal text rendering; extend with a `layer: int` argument that varies output density.
- **`_load_graph()` + `_reload_if_stale()` (`serve.py:308`, ~620)** — graph-load path with mtime reload; branching-factor precomputation attaches here.
- **`_record_traversal()` (`serve.py:102`)** — telemetry hook; `search_strategy` becomes a new field on each record.
- **`_check_derived_edges()` (`serve.py:129`)** — 2-hop A→C inferred edge materialization; provides scaffolding for TOKEN-04 deferred work.
- **`_compute_hot_cold()` (`report.py:16`) + Usage Patterns section (`report.py:219`)** — existing reporting surface surfaces `search_strategy` alongside hot/cold paths with no new file.

### Established Patterns
- **Tools return text wrapped in `types.TextContent`.** D-02 keeps this compatible by making the JSON metadata block trailing content parseable by a sentinel header — preserves the pattern rather than switching to structured MCP content types.
- **`graph.json` is read-only; writes go to sidecars.** Applies to any TOKEN-04 closure cache: must live in `graphify-out/cache/` or similar, never in `graph.json`.
- **`peer_id` defaults to `"anonymous"`.** Continuation token does not include peer identity — query-shape only — so drill-down is peer-neutral.
- **D-13 (v1.1): mtime-driven reload.** Continuation token must hash the mtime so drill-down across a rebuild surfaces `status="graph_changed"` cleanly.

### Integration Points
- **`serve.py` `list_tools()`** — three new parameters on `query_graph` schema (backward-compat defaults).
- **`serve.py` `_tool_query_graph()` call-tool handler** — dispatch on `layer` and `continuation_token`; emit cardinality estimate and search_strategy metadata.
- **`report.py` Usage Patterns** — no change; new `search_strategy` telemetry surfaces automatically since Phase 9.1's aggregation reads telemetry records agnostically.
- **Tests: `tests/test_serve.py`** — extend existing MCP tool tests; add new cases for layer progression, cardinality, bi-BFS, disjoint frontier, continuation_token roundtrip, `graph_changed` invalidation.
- **No changes required in** `extract.py`, `build.py`, `cluster.py`, `analyze.py`, `export.py`.

<specifics>
## Specific Ideas

- "Token economy is THE viability lever for v1.3" — agents will reject graphify if queries blow context budget (per `.planning/notes/april-2026-v1.3-priorities.md`). Phase 9.2 is the (a) priority for that reason. Ship core over breadth.
- Cardinality estimate should feel like "git status before git pull" — cheap, honest, and advisory. Not a gate, not a promise, just information to decide.
- Bidirectional BFS is the single biggest algorithmic win here. Depth-6 unidirectional on a branching-factor-7 graph is 7^6 ≈ 118k node visits; bidirectional meets at depth-3 from each side, 2×7^3 ≈ 686 visits — ~170× reduction.
- Continuation_token should be opaque to the agent but transparent to server-side debugging — base64(JSON) is the right shape. Agents don't parse it; logs do.

</specifics>
