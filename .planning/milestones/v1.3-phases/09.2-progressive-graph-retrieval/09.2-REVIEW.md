# Phase 9.2: Code Review Report

**Reviewed:** 2026-04-16T17:30:00Z
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Phase 9.2 delivers a coherent implementation of progressive graph retrieval across
three feature clusters: (1) token/cardinality estimators + continuation-token codec,
(2) hand-rolled bidirectional BFS + target synthesizer, and (3) layer-aware subgraph
rendering + hybrid response dispatch. The MCP contract (`_tool_query_graph` as
synchronous `-> str`) is preserved. TOKEN-04 scope is respected — no Bloom filter
or materialized closure cache leaked into `serve.py`, and there is an explicit
regression test enforcing this (`test_token_04_deferred_no_bloom_filter_code`).

Primary security invariants hold with one meaningful gap: **`sanitize_label` is
applied consistently to node labels at every layer, but edge-attribute fields
(`relation`, `confidence`) and several L3 node attributes (`source_file`,
`source_location`, `community`) are passed through unsanitized** when rendered
into the output text body. These values originate from `graph.json` and flow
untransformed into the text stream that is returned to the MCP client. That
widens the T-09.2-12 invariant beyond what the test suite currently enforces.

Correctness issues found are subtler: `_estimate_cardinality` can return
`est_nodes == 0` on degenerate inputs (branching factor < 1 with high depth),
causing `estimate_exceeded` to never fire; and `_subgraph_to_text` L2/L3 emit
edges without dedup — acceptable because `_run_query_graph` dedupes at the
telemetry boundary, but rendered text can still show the same edge twice
when bi-BFS records it from both directions.

Continuation-token integrity, size-cap ordering (check before base64 decode),
MCP async/sync boundary, and the `budget` clamp (50..100000) at dispatch head
all look correct. No critical issues were identified.

## Warnings

### WR-01: sanitize_label not applied to edge/node attributes in rendered text

**File:** `graphify/serve.py:670-701`
**Issue:** T-09.2-12 says every label emitted in L1/L2/L3 must go through
`sanitize_label`. The invariant is met for node labels, but the following
attributes are interpolated into the text stream unsanitized:

- L2 + L3 edges: `ed.get('relation', '')` and `ed.get('confidence', '')`
  (serve.py:690-691 and 699-701).
- L3 nodes: `d.get('source_file', '')`, `d.get('source_location', '')`,
  `d.get('community', '')` (serve.py:695-697).
- L1 + L2 nodes: `d.get('community', '')` (serve.py:677, 684).

These values originate from `graph.json`, which Phase 9.1 does not treat as
trusted. A control character, newline, or `---GRAPHIFY-META---` substring
in any of these fields would either corrupt the rendered output or, in the
worst case, inject a spurious meta-sentinel boundary that fools naive
clients into mis-parsing the hybrid response (Pitfall 4).

`test_subgraph_to_text_all_layers_sanitize_labels` only exercises the `label`
field, so the gap is not caught by the current test matrix.

**Fix:**
```python
# Factor a small helper so every emission site is forced through it.
def _clean(val: object) -> str:
    return sanitize_label(str(val)) if val is not None else ""

# L2 edges
lines.append(
    f"EDGE {u_label} --{_clean(ed.get('relation'))} "
    f"[{_clean(ed.get('confidence'))}]--> {v_label}"
)

# L3 nodes — sanitize every interpolated attribute
line = (
    f"NODE {_clean(d.get('label', nid))} "
    f"[src={_clean(d.get('source_file'))} loc={_clean(d.get('source_location'))} "
    f"community={_clean(d.get('community'))}]"
)
```
Add a regression test that seeds `\x00` into `relation`, `confidence`, and
`source_file` and asserts `"\x00"` is absent from every layer's output.

### WR-02: `_estimate_cardinality` can return zero nodes on low-branching graphs, bypassing `estimate_exceeded`

**File:** `graphify/serve.py:629-651`
**Issue:** For `depth >= 2`, `est_nodes` is computed as
`int(len(start_nodes) * branching_factor ** depth)`. If `branching_factor < 1.0`
(sparse/disconnected graphs — e.g. `2*E/N` on a mostly isolated corpus can
easily land at 0.5) and `depth` is large enough, the product rounds down to
`0`. The upper-bound clamp `min(..., G.number_of_nodes())` does not protect the
lower bound, so the function returns `{"nodes": 0, "edges": 0, "tokens": 0}`.

Downstream, `_run_query_graph` uses `cardinality_estimate["tokens"] > 10 * budget`
to gate `estimate_exceeded`. A zero-token estimate always passes the gate,
so the dispatcher will then run the full traversal even though the estimator
genuinely had no opinion — the opposite of the advisory-abort behavior the
estimator is meant to provide. The existing clamp test
(`test_estimate_cardinality_clamped_to_graph_size`) covers overestimation
only.

**Fix:**
```python
else:
    est_nodes = min(
        max(len(start_nodes), int(len(start_nodes) * (branching_factor ** depth))),
        G.number_of_nodes(),
    )
est_edges = min(
    max(0, int(est_nodes * branching_factor / 2)),
    G.number_of_edges(),
)
```
Add a regression test with `branching_factor=0.5, depth=6, start_nodes=["n1"]`
asserting `est_nodes >= 1` so the estimator remains a meaningful
lower bound.

### WR-03: L2/L3 rendering does not dedup edges — bi-BFS meet-in-middle may emit duplicate EDGE lines

**File:** `graphify/serve.py:685-702`
**Issue:** `_bidirectional_bfs` records forward edges as `(u, v)` and reverse
edges as `(neighbor, n)` (explicit "reversed direction marker" at serve.py:877).
When the frontiers meet on a shared node, the meeting edge can appear twice
in `edges_seen` — once from the forward expansion and once from the reverse
expansion — in the two different tuple orderings.

`_run_query_graph` dedupes via `{(min(u,v), max(u,v))}` before calling
`_record_traversal` (serve.py:812), so telemetry counters are correct. But
`_subgraph_to_text` receives the raw (un-deduped) `edges_seen` list and
iterates it directly, so L2 and L3 output can emit two `EDGE` lines for a
single underlying edge — once as `A --rel--> B` and once as `B --rel--> A` —
wasting budget and producing contradictory-looking output on undirected
graphs.

**Fix:** Either dedup in the caller before rendering, or in the renderer
itself:
```python
# In _run_query_graph, pass deduped_edges to the renderer (reuse the dedup done
# for telemetry so budget accounting is consistent):
text_body = _subgraph_to_text(
    G, visited, deduped_edges, token_budget=budget, layer=layer
)

# Or, inside _subgraph_to_text, dedup in place:
seen_pairs: set[tuple[str, str]] = set()
for u, v in edges:
    key = (min(u, v), max(u, v))
    if key in seen_pairs:
        continue
    seen_pairs.add(key)
    ...
```
Note that forwarding `deduped_edges` also changes edge-ordering (the dedup
flattens insertion order). Call it out in the docstring so the test suite
can be updated to match.

### WR-04: `explicit_targets` accepted without size or type validation

**File:** `graphify/serve.py:770-777`
**Issue:** When the MCP client supplies `target_nodes`, the dispatch path
runs `targets = list(explicit_targets)` and passes them straight to
`_bidirectional_bfs`. Two failure modes:

1. If the client passes a large list (say 50 000 entries) the reverse
   frontier is seeded with every one, and `len(reverse_visited)` is already
   past `max_visited = max(10, int(budget/50))` before the first hop. This
   is bounded by the budget check in bi-BFS, so it degrades to
   `budget_exhausted` gracefully — but also means valid large target sets
   never run. More importantly, the server has already allocated memory
   for the full set.
2. If the client passes a non-list (e.g. a string) `list("hello")` silently
   splits into single-character IDs, none of which exist in the graph, and
   bi-BFS quickly returns `frontiers_disjoint`. No error is surfaced.

Neither is a crash or injection, but the schema (see line 952 — JSON
Schema currently does not document `target_nodes` at all; it exists only
as an implicit `arguments.get("target_nodes")`) doesn't tell clients
what is expected.

**Fix:** Either validate and document, or explicitly cap:
```python
raw_targets = arguments.get("target_nodes")
if isinstance(raw_targets, list):
    # Cap at 20 (matches _synthesize_targets upper bound)
    explicit_targets = [str(t) for t in raw_targets[:20]]
else:
    explicit_targets = None
```
And either add `target_nodes` to the schema in `_query_graph_input_schema`
or drop the implicit feature entirely.

## Info

### IN-01: `communities` parameter to `_run_query_graph` is never used

**File:** `graphify/serve.py:727, 1140`
**Issue:** `_run_query_graph` accepts a `communities: dict` parameter and
`_tool_query_graph` passes `communities=communities` on every call, but
the body never reads it. Dead signature surface.
**Fix:** Drop the parameter or add a TODO noting that a future
community-aware renderer will consume it.

### IN-02: Redundant `import math` inside `_synthesize_targets`

**File:** `graphify/serve.py:585`
**Issue:** `math` is already imported at module top (serve.py:8), so the
`import math` at the head of `_synthesize_targets` is dead.
**Fix:** Remove the local import.

### IN-03: Redundant `"" +` concatenation in short-circuit responses

**File:** `graphify/serve.py:745, 758, 772, 782`
**Issue:** All four short-circuit return paths build responses as
`"" + QUERY_GRAPH_META_SENTINEL + json.dumps(...)`. The leading empty
string is a no-op.
**Fix:** `return QUERY_GRAPH_META_SENTINEL + json.dumps(meta, ensure_ascii=False)`.

### IN-04: Sentinel value is textual — no defense if an attribute contains the literal marker

**File:** `graphify/serve.py:715`
**Issue:** `QUERY_GRAPH_META_SENTINEL = "\n---GRAPHIFY-META---\n"` is a bare
string. `sanitize_label` strips control characters but does not strip
arbitrary substrings, so a node label or (per WR-01) an edge relation
containing the literal `---GRAPHIFY-META---` would split the response at
the wrong point.
**Fix:** Either (a) switch to a high-entropy marker (`f"---GRAPHIFY-META-{uuid4().hex}---"`
generated once per response, escape occurrences in the body); (b) return a
structured pair and let `_tool_query_graph` serialize; or (c) add a dedicated
test that a label containing the sentinel literal is handled. Option (b) is
cleanest but breaks the "sync `-> str`" MCP contract noted in CONTEXT; (a)
preserves it.

### IN-05: L3 EDGE rendering does not sanitize edge attributes or verify edge membership

**File:** `graphify/serve.py:686, 698`
**Issue:** Both L2 and L3 emit `G.edges[u, v]` after checking
`u in nodes and v in nodes`, which ensures `u` and `v` are in the rendered
node set but NOT that the edge exists in G. If `edges_seen` is ever out
of sync with G (e.g. after a `_reload_if_stale()` mid-request — which can't
happen today, but is a possible future regression), this raises `KeyError`.
**Fix:** Guard with `G.has_edge(u, v)` or wrap in `try/except KeyError` and
skip. Minor defense-in-depth; no current bug.

### IN-06: Reverse-direction "marker" in bi-BFS is never consumed, producing inconsistent EDGE arrows

**File:** `graphify/serve.py:877`
**Issue:** The reverse expansion appends `edges_seen.append((neighbor, n))`
with a `# reversed direction marker` comment, but nothing downstream
distinguishes forward edges from reverse edges. In L2/L3 rendering, a
reverse-expansion edge therefore appears as
`EDGE {child_label} --relation--> {parent_label}`, which is the "wrong"
direction relative to the forward-expansion convention. The underlying
graph is undirected (`nx.Graph`), so semantics are unchanged — but the
presentation is inconsistent.
**Fix:** Either drop the "marker" comment (it is misleading), or normalize
to forward tuples at the return site:
```python
edges_seen.append((n, neighbor))  # same shape as forward expansion
```

### IN-07: `depth >= 3` path always runs bi-BFS even when `mode="dfs"` is explicitly requested

**File:** `graphify/serve.py:767-783`
**Issue:** The input schema documents `mode` as
`"bfs=broad context, dfs=trace a specific path (depth < 3 only)"`, and
the dispatch correctly ignores `mode` once `depth >= 3` (unconditional
bi-BFS per D-06). Agents that depend on DFS traversal semantics at
depth 3+ will silently get bi-BFS results with no diagnostic.
**Fix:** When `depth >= 3` and `mode == "dfs"`, consider either (a)
emitting a warning status like `"mode_overridden"` in the meta payload, or
(b) updating the `mode` description to say "ignored at depth >= 3".
Currently neither is done, so the behavior is documented only in the
plan's `D-06` decision — not in any user-facing surface.

### IN-08: `token_budget` schema alias lacks `minimum`/`maximum` constraints

**File:** `graphify/serve.py:939-942`
**Issue:** The new `budget` property declares `minimum: 50, maximum: 100000`,
but the deprecated `token_budget` alias omits both. Server-side dispatch
applies the `max(50, min(budget, 100000))` clamp regardless, so this is
a documentation inconsistency, not a security hole. Still, a client
validating against the schema could submit `token_budget: 10**9` and
only discover the limit via clamping.
**Fix:** Mirror the `minimum`/`maximum` on `token_budget` for schema-level
hygiene:
```python
"token_budget": {
    "type": "integer",
    "default": 2000,
    "minimum": 50,
    "maximum": 100000,
    "description": "DEPRECATED - alias for `budget`. Kept for backward compatibility.",
},
```
