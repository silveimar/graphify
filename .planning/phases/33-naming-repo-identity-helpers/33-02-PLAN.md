<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI/profile input -> `resolve_repo_identity()` | User-controlled identity strings become repo slugs used by later filenames, tags, manifests, and sidecars. |
| `.git/config` -> fallback identity | Local repository metadata is parsed without invoking external git commands. |
| profile.yaml -> `validate_profile()` | Vault-owned configuration controls concept naming behavior and repo identity. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-06 | Tampering | `normalize_repo_identity()` | mitigate | Reject explicit path-like identity values containing `/`, `\`, or `..`; normalize to a short slug before any downstream path/tag/manifest use. |
| T-33-07 | Spoofing | repo identity source precedence | mitigate | Resolve once with explicit source strings and stderr reporting so users can see whether CLI, profile, git remote, or cwd won. |
| T-33-08 | Denial of Service | long repo identity values | mitigate | Cap normalized slugs to 80 characters with deterministic hash suffix; reject empty explicit values. |
| T-33-09 | Tampering | `.git/config` parsing | mitigate | Parse with stdlib, extract only the remote basename slug, and fall back to cwd when config is missing/malformed. |
| T-33-10 | Information Disclosure | stderr reporting | accept | Only the normalized repo identity and source are reported; raw remote URLs and full filesystem paths are not required in the success line. |
</threat_model>
