# Phase 59.1: Version sync hygiene and --version flag - Context

**Gathered:** 2026-05-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Resolve the runtime "skill stamp ('X.Y.Z') is older than the installed package" warning that fires on every `graphify` CLI invocation when `pip install -e .[…]` upgrades the package without a follow-up `graphify install`, and enrich `graphify --version` / `-V` output so users can diagnose package/skill version state without grepping stamp files.

**In scope:**
- Replace per-command warning behavior in `_check_skill_version()` (`graphify/__main__.py:21`) with silent auto-self-heal when stamp < package
- Preserve the existing warning shape only for the unusual "stamp newer than package" case (mixed install signal)
- Expand `--version` / `-V` output (`graphify/__main__.py:1335`) from a single line to a multi-line block with package, per-platform stamps, Python version, install mode/path
- Add a `version sync` section to `graphify doctor`
- Mint a new VSYNC-* requirement group in `.planning/REQUIREMENTS.md` and back-fill the Phase 59.1 row in ROADMAP.md
- Tests covering self-heal happy path, write-failure fallback, stamp>package warning preservation, and multi-line `--version` output

**Out of scope (deferred to other phases):**
- `scripts/bump_version.py` changes
- Release-process automation (CI gate enforcing pyproject↔stamp sync, MCP server.json reconciliation)
- New top-level subcommands (`graphify sync-stamps`, `graphify install --refresh-stamp`)
- Opt-out toggles (`GRAPHIFY_NO_HEAL` env, profile flag)
- Machine-readable `--version --json`

</domain>

<decisions>
## Implementation Decisions

### Drift Behavior Policy
- **D-01:** When `_check_skill_version()` detects stamp < package on a regular CLI invocation, it MUST silently rewrite `.graphify_version` to the current package version (`__version__`) and continue. No stderr output on the heal path.
- **D-02:** Self-heal applies to ALL platforms in `_PLATFORM_CONFIG` whose install dir contains an existing `.graphify_version` stamp. Platforms with no stamp file (never installed) remain untouched — preserving today's "no stamp ⇒ no warning" semantics.
- **D-03:** When stamp > package (mixed/downgraded install), the existing two-line stderr warning at `__main__.py:46` is preserved. Do NOT rewrite the stamp downward.
- **D-04:** If the stamp write fails (OSError, PermissionError, missing dir), fall back to the existing two-line warning shape from current `_check_skill_version`. Never abort the user's actual command for a stamp-write failure.
- **D-05:** Skip rules: preserve the current `install` / `uninstall` / `--version` / `-V` / `-h` / `--help` skips at `__main__.py:1344-1346`. Additionally, if `package_version()` cannot resolve (importlib.metadata raises), abort the heal silently — do not write a bogus stamp.
- **D-06:** No opt-out (no env var, no config flag). Heal is unconditional within the rules above.

### `--version` / `-V` Output
- **D-07:** `--version` and `-V` produce a multi-line block. First line: `graphify <pkg-version>`. Subsequent indented lines: `skill stamps:` block with one row per platform (`<platform-name>: <stamp> (<install-dir-with-~>)` for installed, `<platform-name>: — (not installed)` otherwise); `python: <interpreter-version>`; `install: <editable|site-packages> (<resolved-path>)`.
- **D-08:** Drift in `--version` output gets NO visual annotation (no `(drifted)` suffix). Users read the version numbers and infer state. Rationale: `--version` runs with self-heal skipped (read-only diagnostic), so the first post-upgrade `--version` invocation may legitimately show stamp < package; the next regular command silently heals. This is intended.
- **D-09:** `_cli_exit()`'s success-footer at `__main__.py:1370` (`[graphify] version <pkg>`) stays exactly as-is — single line, package version only. Do NOT replicate the multi-line block in the footer.
- **D-10:** `--version` MUST remain side-effect-free: continue to skip self-heal during `--version` / `-V` invocation.

### Sync Mechanism on Detection
- **D-11:** No new top-level subcommand and no new flag on `install`. Auto-heal covers the common case; `graphify install --platform X` covers the manual case as today.
- **D-12:** `graphify doctor` gains a `version sync` section listing each platform with: stamp value, package value, status (`✓ in sync` / `! drifted-newer` / `— not installed`). Mirrors the `--version` block; no heal-history sentinel file (status is computed on the fly).
- **D-13:** Self-heal logic lives inline inside `_check_skill_version()` — replace the current body. No new helper function. Same call site (line 1349 loop), minimal diff surface.

### Release-Process Hygiene Scope
- **D-14:** Phase 59.1 is strictly runtime CLI behavior. `scripts/bump_version.py`, `scripts/sync_mcp_server_json.py`, and CI gating are explicitly out of scope.
- **D-15:** Mint a new **VSYNC-** REQ-ID group in `.planning/REQUIREMENTS.md` for traceability:
  - VSYNC-01 — silent self-heal on stamp < package (D-01..D-06)
  - VSYNC-02 — multi-line `--version` / `-V` output (D-07..D-10)
  - VSYNC-03 — `doctor` version sync section (D-12)
  - VSYNC-04 — preserved warning for stamp > package (D-03)
  Update the v1.12 traceability table to map all four to Phase 59.1.

### Tests in Scope
- **D-16:** Self-heal happy path: write a fake stamp older than `__version__`, run `__main__.main()` with a no-op command, assert stamp file rewritten and stderr is silent.
- **D-17:** Self-heal failure fallback: monkeypatch `Path.write_text` to raise `PermissionError`; assert the existing two-line warning is emitted and command exit code unchanged.
- **D-18:** Stamp-newer warning preserved: write a stamp greater than `__version__`; assert the existing warning fires and the file is NOT overwritten.
- **D-19:** `--version` multi-line snapshot: with two fake platform stamp dirs under `tmp_path`, assert output contains the package line, both stamp rows (with `~`-shortened paths), `python:` line, and `install:` line. Brittle by design — locks the contract.

### Claude's Discretion
- Exact ordering of fields inside the `--version` block (package → stamps → python → install is the current proposal; planner may rearrange if a stronger convention emerges from research).
- Exact wording of the `doctor` section header (e.g., "Version sync" vs "Skill stamps") and per-row formatting.
- Whether to introduce a tiny private helper `_render_version_block()` to share logic between `--version` handling and `doctor` rendering.

<specifics>
## Specific Ideas

- Real-world warning text the user is hitting: `skill stamp ('0.4.7') is older than the installed package; run 'graphify install' to update the skill. (package is '1.0.0')` — this is the exact noise to silence via auto-heal.
- `--version` output is the natural place to surface drift state without users running `doctor`. Multi-line format chosen over `--verbose` opt-in (rejected: full block should be the default, not a hidden mode).
- Self-heal must NOT log a `[graphify] info: refreshed stamp …` line — silence is the desired UX. Audit trail is the stamp file mtime itself.

</specifics>

<deferred>
## Deferred Ideas

- **`--version --json` machine-readable output** — useful for CI parsing but not requested for 59.1; capture if a v1.13+ scope expands to programmatic version querying.
- **`graphify install --refresh-stamp` fast-path** — would skip skill-file copy and only rewrite the stamp; redundant once auto-heal lands.
- **Top-level `graphify sync-stamps` subcommand** — discoverable but unnecessary given silent heal.
- **`GRAPHIFY_NO_HEAL=1` opt-out env var** — useful for CI drift detection but adds a code path; defer until a concrete CI need surfaces.
- **`version.auto_heal: bool` in `.graphify/profile.yaml`** — wrong layer; the profile is for vault adapter behavior, not CLI hygiene.
- **`scripts/bump_version.py` UX nudge** — printing "Run 'graphify install' to refresh skill stamps." after a bump would be friendly but is out of 59.1's strictly-runtime scope.
- **CI gate test asserting `pyproject` ↔ `__version__` ↔ stamp consistency at build time** — preventive instead of reactive; consider in a future "release hygiene" phase.
- **Replace `_cli_exit()` footer with multi-line block** — explicitly rejected (D-09): would be very noisy on every successful command.

</deferred>
