## Drift Behavior Policy

### Q1 — When stamp differs from package on a regular command, what do we do?

| Option | Description | Selected |
|--------|-------------|----------|
| Auto-self-heal silently | Rewrite stamp to current package version; warn only on stamp>package | ✓ |
| Warn once per stamp value | Cache last-warned in sentinel file; warn once per upgrade | |
| Move check to `doctor` only | Remove per-command warning entirely | |
| Keep current behavior | Warn on every command (status quo) | |

**User's choice:** Auto-self-heal silently
**Notes:** Recommended option. Reverse case (stamp newer than package) still warns since it indicates an unusual mixed install.

### Q2 — Self-heal scope: which platforms get auto-rewritten?

| Option | Description | Selected |
|--------|-------------|----------|
| All 7 platforms with existing stamps | Iterate same set as today's drift check; rewrite each stamp where install dir contains `.graphify_version` | ✓ |
| Only the platform matching CWD/active context | Rewrite just the active platform | |
| Only the first-found mismatched stamp | Walk install dirs; rewrite first drift; return | |

**User's choice:** All 7 platforms with existing stamps

### Q3 — Should a self-heal write produce any user-visible signal?

| Option | Description | Selected |
|--------|-------------|----------|
| Silent | No stderr line; mtime is the audit trail | ✓ |
| One-line stderr info | `[graphify] info: refreshed skill stamp X -> Y` | |
| Only when stamp > package | Silent on common path; warn only on reverse drift | |

**User's choice:** Silent

### Q4 — If self-heal write fails, what's the fallback?

| Option | Description | Selected |
|--------|-------------|----------|
| Fall back to current warning | Catch OSError/PermissionError; emit existing two-line warning | ✓ |
| Silent failure — just continue | Swallow exception | |
| Abort with error | Exit 2 | |

**User's choice:** Fall back to current warning

### Q5 — Are there cases where we should NOT self-heal even when stamp < package?

| Option | Description | Selected |
|--------|-------------|----------|
| Skip when stamp is newer (mixed install) | Already implied by recommendation | |
| Skip during `install` / `uninstall` | Already skipped today | |
| Skip when package version cannot be resolved | If importlib.metadata raises, abort heal silently | |
| No additional skip rules | Heal whenever stamp < package and writable | ✓ |

**User's choice:** No additional skip rules
**Notes:** Interpreted to mean no rules ADDITIONAL to those already locked in via the recommended option (which preserves install/uninstall/--version skips and warns on stamp>package). The "skip when package version cannot be resolved" rule was folded into D-05 as a defensive guard since writing a bogus stamp would be worse than skipping.
