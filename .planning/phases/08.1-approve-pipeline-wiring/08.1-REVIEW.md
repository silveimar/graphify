## Summary

This phase wires the `approve` CLI subcommand to the merge engine. The core implementation in `graphify/__main__.py` is clean and well-structured: path confinement is applied before any file write, the indirection helpers (`_load_profile_for_approve`, `_compute_merge_plan_for_approve`, etc.) make monkeypatching in tests straightforward, and the user-modified/SKIP_PRESERVE logic correctly prevents proposal deletion on conflict. ## Warnings

### WR-01: Hardcoded `graphify-out/` manifest path ignores `--out-dir`

**File:** `graphify/__main__.py:843`
**Issue:** `_approve_and_write_proposal` always resolves the vault manifest at the hardcoded relative path `graphify-out/vault-manifest.json`, regardless of the `--out-dir` argument passed on the command line. When a caller uses `--out-dir /some/other/path`, the proposals are loaded from `/some/other/path/proposals/` but the manifest is still read from `./graphify-out/vault-manifest.json`. User-modified detection will miss (or use the wrong) manifest, silently treating all notes as unmodified.

The `proposals_dir` and `out_dir` are already threaded through to the CLI handler (line 1231), but `_approve_and_write_proposal` has no visibility into `out_dir`.

**Fix:** Accept `out_dir: Path` as an additional parameter (or derive it from `proposals_dir.parent`) and resolve the manifest relative to it:

```python
def _approve_and_write_proposal(
    proposals_dir: Path,
    record_id: str,
    vault_path: Path,
    *,
    force: bool = False,
    out_dir: Path | None = None,
) -> dict:
    ...
    _out = out_dir if out_dir is not None else proposals_dir.parent
    manifest_path = _out / "vault-manifest.json"
    old_manifest = _load_manifest_for_approve(manifest_path)
```

The CLI callers at lines 1283 and 1295 would pass `out_dir=out_dir`.

### WR-02: `_approve_and_write_proposal` swallows all merge-engine exceptions in batch-approve path

**File:** `graphify/__main__.py:1285-1287`
**Issue:** In the `--all` batch-approve path, any exception from `_approve_and_write_proposal` is caught by a broad `except Exception` and printed, but the loop continues and exits with code 0 (line 1287). A caller that checks the exit code to confirm all proposals were processed will see 0 even if every approval failed. This can silently lose proposals.

```python
# current (lines 1281-1287)
for p in proposals:
    try:
        _approve_and_write_proposal(...)
        print(f"Approved: ...")
    except Exception as exc:
        print(f"error approving {p['record_id'][:8]}: {exc}", file=sys.stderr)
sys.exit(0)   # always 0
```

**Fix:** Track failure count and exit with code 1 when any approval failed:

```python
failed = 0
for p in proposals:
    try:
        _approve_and_write_proposal(...)
        print(f"Approved: ...")
    except Exception as exc:
        print(f"error approving {p['record_id'][:8]}: {exc}", file=sys.stderr)
        failed += 1
sys.exit(1 if failed else 0)
```

### WR-03: `_reject_proposal` imports `os` inside the function body

**File:** `graphify/__main__.py:759`
**Issue:** `import os as _os` is placed inside `_reject_proposal` rather than at the module level. While not a bug in isolation, the pattern is inconsistent with all other stdlib usage in this file and can mask `ImportError` exceptions at unexpected call sites. More importantly, if the atomic write (`os.replace`) is on a hot path for batch rejection, the repeated import lookup adds minor overhead and looks like an oversight. The module already imports `json`, `re`, `shutil`, `sys`, `Path` at the top.

**Fix:** Move `import os` to the top-level imports (lines 3-8):

```python
import json
import os
import platform
import re
import shutil
import sys
from pathlib import Path
```

Then remove the `import os as _os` on line 759 and replace `_os.replace(tmp, path)` with `os.replace(tmp, path)`.

### IN-01: `_cursor_install` type hint says `Path` but body uses `or Path(".")`

**File:** `graphify/__main__.py:408-410`
**Issue:** The function signature is `def _cursor_install(project_dir: Path) -> None:` but the body immediately applies `(project_dir or Path("."))`, which implies `None` is an accepted input. This contradicts the declared type and will confuse type checkers. The equivalent functions `gemini_install`, `claude_install`, and `gemini_uninstall` all declare `Path | None`.

**Fix:** Align the signature with the actual accepted type:

```python
def _cursor_install(project_dir: Path | None = None) -> None:
```

The same applies to `_cursor_uninstall` at line 422 for consistency.

### IN-02: `test_approve_and_write_proposal_calls_merge` and `test_approve_deletes_proposal` are near-duplicate tests

**File:** `tests/test_approve.py:120-195`
**Issue:** Both tests set up the same four monkeypatches, create a proposal, call `_approve_and_write_proposal`, and assert the proposal file is deleted. The only distinction is that the first also asserts `called["compute"]` and `called["apply"]`. The duplicated boilerplate increases maintenance cost. A shared fixture or parametrized test would reduce this.

**Fix (suggestion):** Extract the monkeypatch boilerplate into a `pytest.fixture` and keep only the distinct assertions in each test. This is a test quality suggestion — the tests as written are correct.

### IN-03: Skill.md Step 6a `validate_profile_preflight` only prints the success literal when there are no errors AND no warnings

**File:** `graphify/skill.md:514-516`
**Issue:** The bash block at lines 514-516 prints the `profile ok — N rules, M templates validated` message only inside `if not result.errors and not result.warnings:`. However, the CLI implementation in `__main__.py` (lines 975-979) prints the ok message whenever there are no errors, even if there are warnings. The skill.md and the CLI are inconsistent: a vault with only warnings will display the ok message at the CLI but nothing from the skill.

**Fix:** Adjust the skill.md bash block to match the CLI's behavior — print the ok message when there are no errors (regardless of warnings):

```bash
if result.errors:
    sys.exit(1)
# D-77a: print ok when no errors (warnings are advisory, not blocking)
print(f'profile ok — {result.rule_count} rules, {result.template_count} templates validated')
sys.exit(0)
```
