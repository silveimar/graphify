<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI path -> filesystem | User-supplied `--input`, `--vault`, and migration artifact paths become local filesystem reads/writes. |
| Vault scan -> preview artifact | Legacy note filenames and frontmatter are untrusted vault content serialized into JSON/Markdown artifacts. |
| Preview artifact -> apply gate | A JSON plan artifact can be stale, edited, or copied from another vault before apply. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-35-01 | Tampering | `scan_legacy_notes()` path handling | mitigate | Resolve the vault root, use bounded `os.walk(..., followlinks=False)`, and validate every candidate with `validate_vault_path()` before adding it to preview rows. |
| T-35-02 | Tampering | `load_migration_plan()` plan id | mitigate | Accept only deterministic SHA-256 digest IDs matching `^[a-f0-9]{16,64}$`, reject path separators and `..`, and recompute the digest from normalized non-volatile preview payload before trusting the artifact. |
| T-35-03 | Tampering | stale or mismatched artifact application | mitigate | `validate_plan_matches_request()` must compare resolved input path, vault path, repo identity, and the content digest against the freshly generated current preview before any apply helper returns applicable actions. |
| T-35-04 | Information Disclosure | preview artifact serialization | mitigate | Store only repo identity, action summaries, relative vault paths, and migration evidence needed for review; do not serialize source file contents or note bodies. |
| T-35-05 | Repudiation | legacy cleanup data loss | mitigate | ORPHAN and legacy `_COMMUNITY_*` rows are review-only, `filter_applicable_actions()` excludes them, and tests assert legacy files still exist after apply. |
</threat_model>
