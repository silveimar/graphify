<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Repo identity source -> render sinks | CLI/profile/git/directory identity reaches filenames, frontmatter, tags, and manifest JSON. |
| Existing manifest -> merge engine | Old and new manifest metadata influences user-modified detection and migration safety. |
| Generated note metadata -> Obsidian | Frontmatter and tags are interpreted by the user's vault tooling. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-35-06 | Tampering | CODE repo frontmatter/tag emission | mitigate | Use `resolve_repo_identity()` from `graphify.naming`, `safe_tag()` for `repo/<identity>`, and `safe_frontmatter_value()` before assigning `frontmatter_fields["repo"]`. |
| T-35-07 | Tampering | manifest repo metadata | mitigate | Record per-note `repo_identity` from rendered frontmatter only after sanitization, and reserve `__graphify_run__` for run-level metadata. |
| T-35-08 | Tampering | old manifest compatibility | mitigate | Treat missing repo identity as unknown and skip reserved `__graphify_` keys during path-entry processing so older manifests preserve user-modified behavior. |
| T-35-09 | Repudiation | repo drift audit | mitigate | Store repo identity at both run and per-note levels so later preview can classify concrete mismatches as SKIP_CONFLICT instead of silently updating another repo's note. |
| T-35-10 | Information Disclosure | manifest JSON | accept | Repo identity is already present in CODE filenames and CLI preview; recording the normalized slug in manifest metadata does not add sensitive source content. |
</threat_model>
