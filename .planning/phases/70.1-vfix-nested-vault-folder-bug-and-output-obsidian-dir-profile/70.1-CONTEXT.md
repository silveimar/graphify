# Phase 70.1 — VFIX: Nested Vault Folder + Output/Profile Path Resolution

## Origin

Bugs surfaced during Phase 70 UAT and follow-up sandbox testing. They are **not** the same as the gaps already closed by Plans 70-08 and 70-09 — those fixed the silent reverse-sync stdout and the auto_on_run input-dir binding. The bugs in this phase are in the path-resolution layer that decides where graphify writes vault notes.

## Bug 1 — Nested Vault Folder

**Symptom (reproduced in `/Users/silveimar/temp-testgph/tmp/uat70-vault/`):**
The vault contains a nested copy of itself:

```
uat70-vault/
├── .graphify/profile.yaml      ← real vault profile
├── .obsidian/
├── People/                     ← real user content
└── uat70-vault/                ← BUG: nested duplicate folder
    └── Atlas/Sources/Graphify/...   ← graphify wrote notes here
```

Expected: notes go directly under `uat70-vault/Atlas/Sources/Graphify/...` (no nested `uat70-vault/`).

**Profile (`.graphify/profile.yaml`) under test:**
```yaml
output:
  exclude: []
  mode: vault-relative
  path: .
taxonomy:
  root: Atlas/Sources/Graphify
graphify_folder_mapping:
  thing: Atlas/Sources/Graphify/Things/
  ...
```

`output.path: .` should mean "vault root." Combined with `--obsidian-dir uat70-vault` (or vault auto-adopt), the writer is concatenating the vault dir name onto itself somewhere in the resolution chain.

## Bug 2 — `--obsidian-dir` × profile `output:` Interaction

When `--obsidian-dir` is provided **and** the profile has an `output:` block, the writer creates an extra nested folder. Hypothesis: the precedence chain at `graphify/__main__.py:1865` (`--output > profile > --obsidian-dir > legacy default`) leaves `obsidian_dir` set as-is when `user_passed_obsidian_dir`, then a downstream join treats it as relative to itself or to vault root after resolved-output already incorporated it.

**Suspect call sites:**
- `graphify/__main__.py:1811-1871` — obsidian_dir resolution + D-08 precedence
- `graphify/output.py` — `resolve_output()` and `ResolvedOutput`
- The line `obsidian_dir = str(resolved.notes_dir)` (lines 1867 & 1869) — does `resolved.notes_dir` already include the vault root, and is it being re-joined with cwd later?
- `graphify/__main__.py:1923` — where `obsidian_dir` is passed into the obsidian writer

## Bug 3 — `cwd` × `--obsidian-dir` × profile `output:` Three-Way Interaction

The user noted "this could be a bug with the cwd vs obsidian-dir vs output config." The writer must produce the same absolute path whether invoked from inside the vault, from the vault's parent, or with an explicit `--obsidian-dir`. Today it does not.

## Scope

In addition to fixing the path-resolution bugs:

1. **Code fix** in `graphify/__main__.py` and/or `graphify/output.py` that produces a single consistent absolute notes path regardless of cwd × `--obsidian-dir` × profile `output:` combinations
2. **Regression tests** covering the matrix: cwd inside vault / cwd parent of vault, with/without `--obsidian-dir`, with/without profile `output.path`
3. **Documentation updates** — README, CLI `--help` text for `--obsidian-dir` and `--output`, the relevant skill files (`graphify/skill*.md`) — describe the precedence rules accurately and include a "common pitfalls" note about nested folders
4. **Profile example files** — the bundled default profile and any `examples/profiles/` files updated to reflect the corrected interpretation of `output.path`, with inline comments clarifying when to set it (and when not to)

## Constraints

- Python 3.10+, pure unit tests, no network or fs side effects outside `tmp_path`
- No new required dependencies
- Backward compatible: existing v1.12 invocations that produced correct paths must keep producing them
- Path confinement (`graphify/security.py`) must not regress — all writes still confined to `graphify-out/` or the resolved vault root
