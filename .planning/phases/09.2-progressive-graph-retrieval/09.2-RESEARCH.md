# Phase 9.2: Progressive Graph Retrieval - Research

**Researched:** 2026-04-16
**Domain:** NetworkX traversal + MCP tool contract extension + deterministic token accounting
**Confidence:** HIGH (stack verified locally: NetworkX 3.4.2 imported; graph.json measured at 2512 nodes / 3926 edges / avg degree 3.13; bidirectional pred/succ helper benchmarked on branching-factor-7 tree)

## Summary

Phase 9.2 extends the existing `query_graph` MCP tool at `graphify/serve.py:463` with three new parameters (`budget`, `layer`, `continuation_token`) and three new helpers (`_bidirectional_bfs()`, `_estimate_cardinality()`, `_subgraph_to_text_layer()`). No new libraries are needed — the Python stdlib (`base64`, `hashlib`, `json`) plus existing NetworkX 3.4.2 cover the entire core scope (TOKEN-01/02/03). The hardest technical question — whether to wrap NetworkX's existing bidirectional helper — is answered clearly: **hand-roll a new `_bidirectional_bfs()` alongside `_bfs()`**, because NetworkX's `_bidirectional_pred_succ` is (a) private (leading underscore, not in the public API), (b) returns pred/succ path dictionaries shaped for path-reconstruction rather than for layer-shaped subgraph rendering, and (c) only terminates on path-meet, not on budget/depth exhaustion — it cannot produce a partial result with `status="frontiers_disjoint"` per D-07. The existing `_bfs()` at `serve.py:350` is the template: a 40-line function returning `(visited: set[str], edges_seen: list[tuple])` — our bidirectional version matches that contract with two frontiers and a meet-point check per layer.

Token accounting uses a **per-layer-shape fixed-cost heuristic**, not tiktoken. Measured node-serialization size on the live `graphify-out/graph.json` is avg 391 bytes / node and 385 bytes / edge; dividing by 4 chars/token (industry approximation matching how `_subgraph_to_text()` already budgets at line 385: `char_budget = token_budget * 3`) gives ~98 tokens for a full-attribute node. CONTEXT.md D-04 specifies Layer 1 ≈ 50 tok/node (id + label + community only), Layer 2 ≈ 200 tok/node (edges + neighbor labels), Layer 3 = full serialization. Because the existing budget logic already uses a chars/token constant, the new layer-density constants stay in the same style — no tokenizer dependency.

Cardinality estimation uses a **1-hop BFS probe + cached average branching factor** (D-04). Average branching factor is computed once in `_load_graph()` as `2 * G.number_of_edges() / G.number_of_nodes()` (measured: 3.13 on the live graph) and attached to module-level state alongside the existing `_graph_mtime`. Estimate for depth-D multi-hop is `seed_count * bf^D` nodes, multiplied by per-layer token density for the token estimate.

**Primary recommendation:** Extend `query_graph` in place per D-01. Hand-roll `_bidirectional_bfs()` matching the `(visited, edges_seen)` contract of `_bfs()`. Add three helpers (`_estimate_cardinality`, `_encode_continuation`, `_decode_continuation`) as module-level functions matching the pattern of `_record_traversal`, `_edge_weight`. All TOKEN-04 stretch work (Bloom filter, materialized closure cache) is already formally deferred per D-09 — do not research further in this phase.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01: Extend the existing `query_graph` MCP tool in place.** Add three new parameters: `budget: int` (total token ceiling for the response), `layer: int` (1|2|3, default 1), and `continuation_token: str` (opaque base64 JSON, optional). Existing callers that pass `token_budget=2000` keep working without behavior change. Old `token_budget` parameter is preserved as a deprecated alias for `budget`; behavior is unified.

- **D-02: Hybrid response format.** Layer N response returns the traditional text summary (current `_subgraph_to_text()` output, extended for layer-specific density) followed by a structured JSON metadata block containing `cardinality_estimate`, `search_strategy`, `status`, `continuation_token`, and `layer`. Keeps backward compatibility with current text-only consumers while giving new consumers machine-readable structure. Parseable by a simple split on a sentinel header.

- **D-03: Stateless drill-down via opaque `continuation_token`.** Layer 1 response includes a base64-encoded JSON token containing original query params, visited frontier, layer progression, and a content-hash of the graph mtime. Agent passes it back on the next call. Server validates the content-hash against current `_reload_if_stale()` mtime; mismatch returns a clean "graph_changed" error so the agent knows to re-query from scratch. No server-side session state.

- **D-04: Exact 1-hop BFS probe with branching-factor extrapolation.** Before any depth ≥ 2 query runs, do a 1-hop BFS from the seed nodes and extrapolate frontier size to requested depth using a precomputed graph-average-branching-factor cached at load time in `_load_graph()`. Cost is O(seed_neighborhood_edges) — milliseconds on 5k-node graphs. Token estimate multiplies node count by per-layer density constants (L1=50, L2=200, L3=full-attr-sum).

- **D-05: Cardinality always returned, abort is advisory.** The `cardinality_estimate` block returns on every multi-hop call regardless of whether the query executes. The agent decides based on its own budget math whether to proceed; the server never refuses on estimate alone (avoids false-positive rejections on under-estimated cheap queries). Exception: if estimate > 10x the supplied `budget`, server returns the estimate + empty result + `status="estimate_exceeded"` without running the full traversal.

- **D-06: Auto-activate bidirectional BFS at depth ≥ 3 unconditionally.** No flag, no heuristic. Matches requirement TOKEN-03 literally and keeps telemetry predictable. When no explicit target is supplied, the "target set" is synthesized from remaining high-degree frontier nodes (top-K by degree, K proportional to graph size). New function `_bidirectional_bfs()` added alongside existing `_bfs()` (line 350) — old function stays as the single-endpoint path for depth < 3.

- **D-07: Disjoint-frontier handling returns partial with structured status.** When the two frontiers don't meet within the budget/depth cap, return whatever was traversed from both sides with `metadata.status="incomplete"` and `metadata.reason="frontiers_disjoint"` (or `"budget_exhausted"`). Include the explored frontier node IDs so the agent can decide whether to retry with larger budget. No auto-fallback to unidirectional — that would silently change `search_strategy` mid-query and muddy telemetry.

- **D-08: `search_strategy` field plugs into existing `_record_traversal()` telemetry.** Added as a new field on each traversal record at `serve.py:102`. Values: `"bfs"`, `"dfs"`, `"bidirectional"`. Surfaces in the "Usage Patterns" section of `GRAPH_REPORT.md` added in Phase 9.1 — no new reporting surface, unified observability.

- **D-09: TOKEN-04 fully deferred.** Bloom filter + materialized 2–3-hop closure cache move to a follow-up phase. Scaffolding exists in `_check_derived_edges()` at `serve.py:129`.

### Pattern 2: Hand-rolled `_bidirectional_bfs()` matching `_bfs()` contract

**What:** New function returning the same `(visited: set[str], edges_seen: list[tuple])` tuple that `_bfs()` returns, so `_subgraph_to_text()` and `_record_traversal()` consume it uniformly.

**When to use:** For depth ≥ 3 per D-06. The forward and reverse frontiers alternate; whichever is smaller expands first (same heuristic NetworkX's private `_bidirectional_pred_succ` uses — [VERIFIED: `inspect.getsource`]).

**Example:**
```python
# graphify/serve.py — new function, placed between _bfs() (L350) and _dfs() (L366)
# Source: adapted from networkx.algorithms.shortest_paths.unweighted._bidirectional_pred_succ
#         (private API; logic re-implemented rather than imported per rationale above)
def _bidirectional_bfs(
    G: nx.Graph,
    start_nodes: list[str],
    target_nodes: list[str],
    depth: int,
    max_visited: int,
) -> tuple[set[str], list[tuple], str]:
    """Meet-in-the-middle BFS. Depth is the total hop budget (forward + reverse combined).

    Returns (visited, edges_seen, status) where status is one of:
      "ok"                 — frontiers met, traversal complete
      "frontiers_disjoint" — exhausted depth/visited without meeting
      "budget_exhausted"   — hit max_visited cap before meet
    """
    forward_visited: set[str] = set(start_nodes)
    reverse_visited: set[str] = set(target_nodes)
    forward_frontier: set[str] = set(start_nodes)
    reverse_frontier: set[str] = set(target_nodes)
    edges_seen: list[tuple] = []

    hops_forward = 0
    hops_reverse = 0
    # Distribute depth budget: half forward, half reverse (round up forward)
    max_forward = (depth + 1) // 2
    max_reverse = depth // 2

    while forward_frontier and reverse_frontier:
        # Termination conditions per D-07
        if len(forward_visited) + len(reverse_visited) >= max_visited:
            return forward_visited | reverse_visited, edges_seen, "budget_exhausted"
        if forward_visited & reverse_visited:
            return forward_visited | reverse_visited, edges_seen, "ok"

        # Expand smaller frontier first (same heuristic as NetworkX)
        if len(forward_frontier) <= len(reverse_frontier) and hops_forward < max_forward:
            next_f: set[str] = set()
            for n in forward_frontier:
                for neighbor in G.neighbors(n):
                    if neighbor not in forward_visited:
                        next_f.add(neighbor)
                        edges_seen.append((n, neighbor))
                        if neighbor in reverse_visited:  # meet
                            forward_visited |= next_f
                            return forward_visited | reverse_visited, edges_seen, "ok"
            forward_visited |= next_f
            forward_frontier = next_f
            hops_forward += 1
        elif hops_reverse < max_reverse:
            next_r: set[str] = set()
            for n in reverse_frontier:
                for neighbor in G.neighbors(n):
                    if neighbor not in reverse_visited:
                        next_r.add(neighbor)
                        edges_seen.append((neighbor, n))  # reversed direction marker
                        if neighbor in forward_visited:
                            reverse_visited |= next_r
                            return forward_visited | reverse_visited, edges_seen, "ok"
            reverse_visited |= next_r
            reverse_frontier = next_r
            hops_reverse += 1
        else:
            break

    return forward_visited | reverse_visited, edges_seen, "frontiers_disjoint"
```

**[VERIFIED via benchmark]** On `networkx.balanced_tree(branching=7, height=6)` (N=137,257, E=137,256), bidirectional traversal from root to leaf visited **74 nodes** in 0.0002s, vs a full unidirectional depth-6 BFS which would visit ~137,000. Confirms the CONTEXT claim of ~170× reduction is conservative (~1800× observed on this topology).

### Pattern 3: Opaque continuation_token (D-03)

**What:** Base64(JSON) payload containing `{query_params, visited_frontier, current_layer, mtime_hash}`. Opaque to agent but debuggable server-side (base64 is trivially decodable in logs).

**When to use:** Every Layer 1/2 response that supports drill-down to the next layer.

**Example:**
```python
# graphify/serve.py — new module-level helpers placed near _record_traversal (L102)
# Source: stdlib-only; design matches CONTEXT D-03 and Claude's discretion note
def _encode_continuation(
    query_params: dict,
    visited: set[str],
    current_layer: int,
    graph_mtime: float,
) -> str:
    """Encode opaque drill-down token. Base64(JSON). Integrity: sha256 of mtime+params."""
    import base64
    import hashlib
    mtime_hash = hashlib.sha256(f"{graph_mtime}:{json.dumps(query_params, sort_keys=True)}".encode()).hexdigest()[:16]
    payload = {
        "q":       query_params,          # original question, depth, mode
        "v":       sorted(visited),       # deterministic — reproducible in tests
        "l":       current_layer,
        "h":       mtime_hash,
    }
    return base64.urlsafe_b64encode(json.dumps(payload, sort_keys=True).encode()).decode()


def _decode_continuation(token: str, graph_mtime: float) -> tuple[dict, str]:
    """Decode + validate. Returns (payload_dict, status). Status in {"ok","graph_changed","malformed"}."""
    import base64
    import hashlib
    try:
        payload = json.loads(base64.urlsafe_b64decode(token.encode()).decode())
    except (ValueError, json.JSONDecodeError):
        return {}, "malformed"
    expected = hashlib.sha256(
        f"{graph_mtime}:{json.dumps(payload.get('q', {}), sort_keys=True)}".encode()
    ).hexdigest()[:16]
    if payload.get("h") != expected:
        return payload, "graph_changed"
    return payload, "ok"
```

### Anti-Patterns to Avoid

- **Importing `_bidirectional_pred_succ` from NetworkX internals.** Private API (leading underscore); breaks on minor version upgrades. Hand-roll the logic matching our `(visited, edges_seen)` contract [VERIFIED by reading `inspect.getsource`].
- **Adding `tiktoken` as an optional dependency just to count tokens accurately.** CONTEXT D-04 and D-05 explicitly chose a heuristic approach; the existing `_subgraph_to_text()` already uses chars/token = 3 at `serve.py:385`. Consistency over accuracy here.
- **Server-side session state keyed by `session_id` for drill-down.** Rejected in DISCUSSION-LOG Q2 of "Tool Contract Shape" — conflicts with v1.1 D-13 mtime-reload pattern.
- **Silently falling back from bidirectional to unidirectional on disjoint frontiers.** D-07 explicitly forbids this — telemetry must stay honest. Return partial with `status="frontiers_disjoint"` instead.
- **Running the full traversal THEN checking whether estimate exceeded budget.** D-05 says estimate must be pre-flight: run the 1-hop probe FIRST, short-circuit with `status="estimate_exceeded"` when estimate > 10× budget.
- **Rendering any LLM-sourced string (node label, edge relation) without `sanitize_label()`.** The existing `_subgraph_to_text()` applies `sanitize_label(d.get("label", nid))` at L386 on every label emission — new layer-shaped paths must do the same.

## Common Pitfalls

### Pitfall 1: NetworkX private API coupling
**What goes wrong:** Copy-pasting `_bidirectional_pred_succ` or importing it directly gives short-term wins but silently breaks on NetworkX minor upgrades because private names have no stability guarantee.
**Why it happens:** The private helper looks perfect — it already does meet-in-the-middle BFS with both-frontier management.
**How to avoid:** Hand-roll `_bidirectional_bfs()` matching our `(visited, edges_seen)` contract. Use NetworkX source as a design reference, not a runtime dependency.
**Warning signs:** A test import like `from networkx.algorithms.shortest_paths.unweighted import _bidirectional_pred_succ` in the code or in `requirements*.txt` — either is a red flag.

### Pitfall 2: Cardinality estimate drift across graph topology
**What goes wrong:** Average branching factor on a star-shaped graph (one hub, many leaves) is misleading — depth-3 estimate will wildly over-predict when a seed is a leaf and wildly under-predict when a seed is the hub.
**Why it happens:** `2*E/N` is a uniform prior; real graphs cluster.
**How to avoid:** D-04 accepts this; D-05 keeps the estimate advisory. Add a unit test that exercises a star-topology + verifies the estimate is returned (not that it's accurate). Document in the inline comment on `_estimate_cardinality()` that accuracy degrades on power-law topologies.
**Warning signs:** UAT reports "the abort recommendation was wrong" — at which point planner decides whether to upgrade to degree-aware extrapolation (deferred discretion note).

### Pitfall 3: Continuation_token integrity check failing on same-graph reload
**What goes wrong:** A legitimate Layer 1 → Layer 2 drill-down fires `status="graph_changed"` because the mtime changed between calls even though the graph content is identical (e.g., filesystem touched the file, git checkout+restore).
**Why it happens:** Mtime is monotonic per file modification, not content-aware.
**How to avoid:** Use mtime hash — not file-content hash — per D-03 (CONTEXT explicitly says "content-hash of the graph mtime"). Accept that a touched-file triggers re-query; it's the safe failure mode (false positive on the "graph changed" detection > false negative on stale data).
**Warning signs:** A complaint-shaped issue in the v1.3 UAT: "my session lost drill-down after `graphify run` even though the graph is identical." Fix at the graphify-rebuild-path level, not the query path — don't try to detect "same content, new mtime."

### Pitfall 4: Backward-incompatible response format
**What goes wrong:** Old MCP client (still passing `token_budget=2000`) receives `text + SENTINEL + JSON` and fails to render it cleanly.
**Why it happens:** The old contract returned plain text in `types.TextContent`.
**How to avoid:** The sentinel (`---GRAPHIFY-META---`) must be emitted on a line by itself. Old clients that just display text see a trailing JSON blob visually — not a crash. New clients split on the sentinel. Include a test that verifies `old_client_parse(response)` (naive text display) still produces valid-looking output.
**Warning signs:** UAT reports garbled output — add the sentinel on its own line with newlines on both sides.

### Pitfall 5: Target-set synthesis for bi-BFS when agent didn't supply a target
**What goes wrong:** Agent calls with just `depth=4` and no explicit target. Bi-BFS needs two endpoints. If the "target set" is empty, bi-BFS degenerates to single-endpoint and telemetry falsely records `search_strategy="bidirectional"`.
**Why it happens:** CONTEXT says "when no explicit target is supplied, the 'target set' is synthesized from remaining high-degree frontier nodes (top-K by degree, K proportional to graph size)" — the synthesizer must never return empty.
**How to avoid:** Target synthesizer: `K = max(3, min(20, int(log2(N))))`; if still empty after filtering out the start set, DO fall back to unidirectional and record `search_strategy="bfs"` (not `"bidirectional"`) — telemetry honesty wins per D-07.
**Warning signs:** `telemetry.json` shows bidirectional records on graphs with < 5 nodes — synthesizer bypass malfunction.

### Pitfall 6: Double-counting telemetry in bi-BFS
**What goes wrong:** Both forward and reverse frontiers touch the same edge; `_record_traversal()` increments it twice in one query call.
**Why it happens:** The forward pass and reverse pass maintain separate edge lists; if both surface the meeting edge, it's recorded twice.
**How to avoid:** Normalize edges to `(min, max)` tuples in `edges_seen` — `_record_traversal` at `serve.py:102` ALREADY normalizes to `f"{min(u,v)}:{max(u,v)}"` on line 107 — so double-adding would not cause a second counter entry but WOULD inflate that counter by 2 per query. Dedupe `edges_seen` before passing to `_record_traversal` via `list(set((min(u,v), max(u,v)) for u,v in edges_seen))`.
**Warning signs:** Phase 9.1's hot-path report shows the meet-edge 2× as hot as its true traversal rate — fix in the BFS function, not the telemetry recorder.

### Cardinality estimation via 1-hop probe (D-04)

```python
# graphify/serve.py — new helper
# Source: CONTEXT D-04; stdlib only
def _estimate_cardinality(
    G: nx.Graph,
    start_nodes: list[str],
    depth: int,
    layer: int,
    branching_factor: float,
) -> dict:
    """1-hop BFS probe + geometric extrapolation.

    Returns {"nodes": int, "edges": int, "tokens": int}.
    Accuracy degrades on power-law graphs (see Pitfall 2).
    """
    if depth <= 0:
        return {"nodes": len(start_nodes), "edges": 0, "tokens": _estimate_tokens_for_layer(len(start_nodes), 0, layer)}
    one_hop_visited, one_hop_edges = _bfs(G, start_nodes, 1)
    one_hop_count = len(one_hop_visited)
    # Geometric extrapolation: bf^depth, clamped at graph size
    if depth == 1:
        est_nodes = one_hop_count
    else:
        est_nodes = min(
            int(len(start_nodes) * (branching_factor ** depth)),
            G.number_of_nodes(),
        )
    # Edge estimate: each new node brings ~branching_factor/2 new edges (undirected)
    est_edges = min(int(est_nodes * branching_factor / 2), G.number_of_edges())
    return {
        "nodes":  est_nodes,
        "edges":  est_edges,
        "tokens": _estimate_tokens_for_layer(est_nodes, est_edges, layer),
    }
```

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | MCP stdio server has no auth surface; trust boundary is process-level (peer_id = "anonymous" default; never derived from environment per STATE.md) |
| V3 Session Management | yes | **Stateless via continuation_token.** D-03 mandates opaque base64(JSON) with mtime-hash integrity check; no server-side session state; `peer_id` defaults to `"anonymous"`; continuation_token does NOT include peer identity (query-shape only) |
| V4 Access Control | no | Single-tenant stdio server; no per-user ACL in scope |
| V5 Input Validation | yes | **All label-shaped string output goes through `graphify.security.sanitize_label`** (HTML-escape, control-char strip, 256-char cap) — already applied at `serve.py:386` in `_subgraph_to_text()`; new layer paths must apply identically. Schema validation of parsed continuation_token: `.get()` + try/except on `json.loads` + type checks before use |
| V6 Cryptography | yes (integrity, not confidentiality) | **`hashlib.sha256` via stdlib** for continuation_token mtime hash (first 16 hex chars for short form). Never hand-roll; no homegrown hash construction. Integrity purpose only — no secret material. |
| V14 Configuration | yes | No new env vars or config keys in scope; graph.json remains read-only per v1.1 contract |

### Known Threat Patterns for stdlib-only MCP-stdio stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Malicious or malformed continuation_token (Tampering) | Tampering | Validate base64 decode wrapped in try/except; validate JSON decode wrapped in try/except; verify mtime hash equals expected before trusting any payload field |
| Node/edge label injection of control chars / HTML into rendered output | Tampering / Information Disclosure | `sanitize_label()` applied on every emitted label — reuse existing pattern at `serve.py:386` |
| Denial-of-Service via huge `continuation_token` payload (oversized `visited` set) | DoS | Token payload is base64(JSON); cap raw token length to a hard ceiling (e.g., 64 KB) before decode; reject oversized tokens with `status="malformed"` |
| Denial-of-Service via `budget=999999999` triggering large synthesized target-set scan | DoS | Clamp `budget` to a sane maximum (existing pattern: `depth = min(int(arguments.get("depth", 3)), 6)` at `serve.py:633`); apply same clamp to `budget` and `K` in target-set synthesis |
| Stale-data read (Information Disclosure) | Information Disclosure | `_reload_if_stale()` called at the top of `_tool_query_graph` (existing at `serve.py:632`); continuation_token mtime hash surfaces `status="graph_changed"` on drift |
| Replay of continuation_token across rebuilds | Tampering | mtime-hash check per D-03 guarantees a fresh rebuild invalidates old tokens cleanly |

All mitigations above are either already present in `serve.py` or are stdlib-native (hashlib, base64, try/except, min()). No new security library or new scanner in scope.
