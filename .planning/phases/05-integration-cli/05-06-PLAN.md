## D-78 (gap closure) — Reinterpret D-73: thin finished-graph utilities are allowed in `__main__.py`

**Status:** New decision, retroactive to Phase 5. Documented here (not in 05-CONTEXT.md) because CONTEXT.md is a historical snapshot and this reinterpretation arose from the gaps surfaced by 05-VERIFICATION.md.

**What D-78 decides:**
The CLI (`graphify/__main__.py`) may host thin top-level flags that operate over:

1. an **already-built `graph.json`** on disk (loaded via the same pattern the `query` command uses), or
2. a **vault directory** read-only (for validation).

Specifically, `--validate-profile <vault>` and `--obsidian [--graph ...] [--obsidian-dir ...] [--dry-run]` are added to the dispatch table. Both are **one-shot library wrappers**: they do not run extract → build → cluster themselves; they consume artifacts the skill has already produced (or, in the validate case, simply read a vault directory).

**Why this closes PROF-05 and MRG-03 without invalidating D-73:**
D-73's original rationale was to prevent porting the extract → build → cluster → analyze → report orchestration into `__main__.py`. Nothing in this plan does that. `--validate-profile` never touches the pipeline; it just runs `validate_profile_preflight(vault)`. `--obsidian` loads a pre-existing `graphify-out/graph.json` (exactly like `query` does) and calls `to_obsidian(G, communities, ...)` which is a library function. The skill remains the pipeline driver. Non-skill users who run `graphify --obsidian` must first produce `graph.json` via the skill or by calling the Python API directly; if no `graph.json` exists, the CLI errors out with an actionable message telling them so.

**Scope boundary (what D-78 does NOT permit):**
- No `graphify run` / `graphify extract` / `graphify build` / `graphify cluster` subcommand — those remain deferred per CONTEXT.md `<deferred>` section.
- No LLM-based extraction path decisions in `__main__.py`.
- No reimplementation of Phase 1–4 logic; the CLI branches are pure glue (argv parsing → library call → print → exit code).
- The pipeline runner deferral (`<deferred>` #1) still stands: "A real `__main__.py` entry that drives extract → build → cluster → export without going through the skill" remains out of scope.

**Traceability:**
- Closes gap truth 1 (PROF-05) from 05-VERIFICATION.md
- Closes gap truth 2 (MRG-03) from 05-VERIFICATION.md
- Preserves SC-3 (MRG-05, already SATISFIED) by routing the CLI `--obsidian` branch through the same `to_obsidian()` that `test_integration.py` exercises — the CLI code path is backward-compatible by construction because it is a thin wrapper over the verified library call.

```python
class PreflightResult(NamedTuple):
    errors: list[str]
    warnings: list[str]
```

Key preflight behaviors (verified by reading `graphify/profile.py` L463–520):
- Non-existent `vault_dir` → `errors=["vault_dir does not exist: <path>"]`, exit 1
- `vault_dir` is a file → `errors=["vault_dir is not a directory: <path>"]`, exit 1
- `vault_dir` exists but has no `.graphify/` subdir → `PreflightResult([], [], 0, 0)`, exit 0
- Malformed `.graphify/profile.yaml` → errors populated, exit 1
- Clean → `PreflightResult([], [], N, M)`, exit 0

  ```python
      # Thin wrapper over graphify.export.to_obsidian. Loads an already-built
      # graph.json from disk (same pattern as the `query` command), reconstructs
      # `communities: dict[int, list[str]]` from node.community attributes, and
      # calls to_obsidian. Never runs extract/build/cluster. The skill is still
      # the pipeline driver for non-standalone use.
      if cmd == "--obsidian":
          graph_path = "graphify-out/graph.json"
          obsidian_dir = "graphify-out/obsidian"
          dry_run = False
          args = sys.argv[2:]
          i = 0
          while i < len(args):
              if args[i] == "--graph" and i + 1 < len(args):
                  graph_path = args[i + 1]; i += 2
              elif args[i].startswith("--graph="):
                  graph_path = args[i].split("=", 1)[1]; i += 1
              elif args[i] == "--obsidian-dir" and i + 1 < len(args):
                  obsidian_dir = args[i + 1]; i += 2
              elif args[i].startswith("--obsidian-dir="):
                  obsidian_dir = args[i].split("=", 1)[1]; i += 1
              elif args[i] == "--dry-run":
                  dry_run = True; i += 1
              else:
                  print(f"error: unknown --obsidian option: {args[i]}", file=sys.stderr)
                  sys.exit(2)

          # Load graph.json — reuse the exact query-command pattern (L805–822).
          gp = Path(graph_path).resolve()
          if not gp.exists():
              print(f"error: graph file not found: {gp}", file=sys.stderr)
              print("hint: run /graphify to produce graphify-out/graph.json first", file=sys.stderr)
              sys.exit(1)
          if gp.suffix != ".json":
              print("error: graph file must be a .json file", file=sys.stderr)
              sys.exit(1)
          try:
              import json as _json
              from networkx.readwrite import json_graph
              _raw = _json.loads(gp.read_text(encoding="utf-8"))
              try:
                  G = json_graph.node_link_graph(_raw, edges="links")
              except TypeError:
                  G = json_graph.node_link_graph(_raw)
          except Exception as exc:
              print(f"error: could not load graph: {exc}", file=sys.stderr)
              sys.exit(1)

          # Reconstruct communities dict from node["community"] attribute
          # (written by to_json at graphify/export.py L293). Nodes with a
          # None community (isolates) are skipped — they would otherwise land
          # in a phantom community -1 that to_obsidian's classify() doesn't
          # expect.
          communities: dict[int, list[str]] = {}
          for node_id, data in G.nodes(data=True):
              cid = data.get("community")
              if cid is None:
                  continue
              try:
                  cid_int = int(cid)
              except (TypeError, ValueError):
                  continue
              communities.setdefault(cid_int, []).append(node_id)

          # Call the library — profile discovery happens inside to_obsidian
          # itself (load_profile(out) falls back to _DEFAULT_PROFILE when the
          # vault has no .graphify/profile.yaml). The CLI passes profile=None.
          from graphify.export import to_obsidian
          from graphify.merge import MergePlan, format_merge_plan
          try:
              result = to_obsidian(
                  G,
                  communities,
                  obsidian_dir,
                  dry_run=dry_run,
              )
          except Exception as exc:
              print(f"error: to_obsidian failed: {exc}", file=sys.stderr)
              sys.exit(1)

          if isinstance(result, MergePlan):
              # Dry-run path: print the full grouped plan via the Phase 5 D-76 helper
              print(format_merge_plan(result), end="")
          else:
              # Normal run: print a one-line summary from MergeResult.plan.summary
              summary = getattr(getattr(result, "plan", None), "summary", {}) or {}
              total = sum(summary.values()) if summary else 0
              created = summary.get("CREATE", 0)
              updated = summary.get("UPDATE", 0)
              print(
                  f"wrote obsidian vault at {Path(obsidian_dir).resolve()} "
                  f"\u2014 {total} actions ({created} CREATE, {updated} UPDATE)"
              )
          sys.exit(0)
  ```

  ```python
  from __future__ import annotations

  import json
  import subprocess
  import sys
  from pathlib import Path

  import networkx as nx

  # ---------------------------------------------------------------------------
  # Fixture builders

  def _run_cli(*args: str) -> subprocess.CompletedProcess[str]:
      """Invoke `python -m graphify <args...>` and return the completed process.

      Uses check=False so the caller owns exit-code assertions.
      """
      return subprocess.run(
          [sys.executable, "-m", "graphify", *args],
          capture_output=True,
          text=True,
          check=False,
      )


  def _write_fixture_graph_json(path: Path) -> None:
      """Build a minimal 4-node / 2-community graph and write it to `path`
      via the real graphify.export.to_json so the community attribute shape
      matches production.
      """
      from graphify.export import to_json

      G = nx.Graph()
      G.add_node("n1", label="Transformer", file_type="code", source_file="model.py")
      G.add_node("n2", label="Attention", file_type="code", source_file="model.py")
      G.add_node("n3", label="LayerNorm", file_type="code", source_file="model.py")
      G.add_node("n4", label="SGD Optimizer", file_type="code", source_file="train.py")
      G.add_edge("n1", "n2", relation="contains", confidence="EXTRACTED", source_file="model.py", weight=1.0)
      G.add_edge("n1", "n3", relation="contains", confidence="EXTRACTED", source_file="model.py", weight=1.0)
      G.add_edge("n2", "n3", relation="uses", confidence="EXTRACTED", source_file="model.py", weight=1.0)
      communities = {0: ["n1", "n2", "n3"], 1: ["n4"]}
      path.parent.mkdir(parents=True, exist_ok=True)
      to_json(G, communities, str(path))
  ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| argv → `--validate-profile` branch | User-supplied `<vault-path>` crosses into `Path(vault_dir)` and `validate_profile_preflight()`. Read-only. |
| argv → `--obsidian` branch | User-supplied `<graph-path>`, `<obsidian-dir>` cross into `Path.resolve()`, `json.loads`, `to_obsidian`. Mixed read + write. |
| graph.json content → `to_obsidian` | Attacker-supplied JSON deserialized via `json.loads` → `networkx.json_graph.node_link_graph`. Same attack surface as the existing `query` command. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-06-01 | Tampering | `--validate-profile <vault>` argv | accept | Read-only operation. `validate_profile_preflight` never writes, never runs code from the vault, never imports Python from the vault. Uses `yaml.safe_load` (not `yaml.load`), established in Phase 1. CLI does zero new validation; it trusts the library function. |
| T-05-06-02 | Information Disclosure | `--validate-profile` stderr output | accept | Error messages include the resolved vault path. This is intentional (actionable messaging per ROADMAP SC-1: "pass/fail with actionable messages"). No secrets crossed — the branch never reads environment variables or credentials. |
| T-05-06-03 | Tampering | `--obsidian --graph <path>` path traversal | mitigate | The CLI branch calls `Path(graph_path).resolve()` and checks `.exists()` + `.suffix == ".json"` (identical to the existing `query` command at `__main__.py` L805–811). `json.loads` is used with a size-unbounded `read_text` — same as `query`. No new attack surface; the existing `query` threat model covers this path. Attacker-supplied path can read any `.json` file on disk the user already has read access to; this is not a CLI-layer concern, it is inherent to the user running `graphify` on their own box. |
| T-05-06-04 | Tampering | `--obsidian --obsidian-dir <path>` write target | mitigate | The CLI branch passes `obsidian_dir` straight through to `to_obsidian`, which internally calls `out.mkdir(parents=True, exist_ok=True)` and then routes every write through `compute_merge_plan` → `apply_merge_plan` → `validate_vault_path` (Phase 1 D-06, enforced in Phase 4 D-71). Path confinement is the library's responsibility and is already tested. The CLI does NOT re-validate — defense-in-depth would be duplicative and risks drift. |
| T-05-06-05 | Denial of Service | Malicious graph.json (billion-laughs, cyclic references, huge node count) | accept | Same as `query` command. `json.loads` is Python stdlib and safe against entity expansion (no entities in JSON). Huge graphs would make `to_obsidian` slow but are already a concern for every existing exporter. No new DoS surface. |
| T-05-06-06 | Tampering | Node label injection via graph.json → rendered `.md` frontmatter | mitigate | Node labels coming out of an attacker-crafted graph.json flow into `render_note` → `safe_frontmatter_value`, `safe_filename`, `safe_tag` (Phase 1 D-06–D-10, Phase 2 D-18). Phase 2 established there is no Jinja2, no `eval`, no `exec`. The CLI branch inherits this guarantee. |
| T-05-06-08 | Information Disclosure | Dry-run formatter `format_merge_plan` leaking paths to stdout | accept | The dry-run output includes full vault paths. This is the intentional behavior of D-76 ("per-file list of what will be written"). Pipes to stdout, not stderr, so redirecting works normally. The user is the one who invoked the command against their own vault; no privilege boundary is crossed. |
