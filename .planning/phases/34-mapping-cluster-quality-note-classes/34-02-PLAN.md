<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| graph node metadata -> mapping context | Node labels, file types, source files, and community IDs are untrusted extraction output. |
| mapping context -> export/templates | `ClassificationContext` drives filenames, frontmatter, tags, links, and vault writes downstream. |
| legacy community semantics -> MOC routing | `community` must not produce `_COMMUNITY_*` overview output. |
| existing vault files -> phase scope | Existing legacy files are outside Phase 34 and must not be scanned or deleted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-02-01 | Tampering | `graphify/mapping.py` node label/source metadata | mitigate | Store only raw classification context; leave filename/frontmatter/wikilink sanitization to existing downstream helpers, and do not construct paths in mapping. |
| T-34-02-02 | Spoofing | CODE eligibility | mitigate | Require `file_type == "code"`, non-empty `source_file`, god-node membership, and synthetic-node exclusions before assigning `note_type: "code"`. |
| T-34-02-03 | Tampering | legacy `community` routing | mitigate | Emit only MOC contexts in `per_community`; no new `render_community_overview()` path or `_COMMUNITY_*` filename logic belongs in mapping. |
| T-34-02-04 | Denial of Service | CODE member lists | mitigate | Cap `code_members` and `code_member_labels` to exactly 10, sorted deterministically by degree then label. |
| T-34-02-05 | Tampering / Data loss | legacy vault files | mitigate | Do not inspect existing vault files for `_COMMUNITY_*`; Phase 35 owns migration visibility per D-04. |
| T-34-02-06 | Spoofing / Collision | CODE filename suffixes | transfer | Mapping exposes node id/source file context only; Plan 03 must group colliding CODE stems and suffix every colliding member deterministically. |
</threat_model>
