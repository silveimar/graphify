# Phase 60.1: update-vault-apply-determinism-fix - Research

**Researched:** 2026-05-03
**Domain:** Community-detection determinism in `graphify/cluster.py` (Leiden seed plumbing)
**Confidence:** HIGH

## Summary

Phase 60.1 is a one-line determinism fix. `graspologic.partition.leiden()` is called without `random_seed` in `graphify/cluster.py:_partition`, so its partition output varies across separate Python processes. That non-determinism propagates through `cluster()` Ôćĺ `migration._digest_payload()` Ôćĺ `compute_migration_plan_id()` and causes `update-vault --apply --plan-id <id>` to fail with "stale or mismatched migration plan" because preview and apply (run as separate subprocesses) hash different action lists. The Louvain fallback in the same file already passes `seed=42`; the Leiden path is the lone outlier.

The fix is locked in CONTEXT.md: change `leiden(G)` to `leiden(G, random_seed=42)` at `graphify/cluster.py:38`. The graspologic 3.4.4 API installed in this environment confirms `random_seed: Optional[int] = None` as the parameter name and default. Because the same `_partition()` helper is invoked again from `_split_community()` (cluster.py:114), the single edit covers both Leiden call paths ÔÇö no second site to change.

**Primary recommendation:** Land the one-line change in `graphify/cluster.py:_partition`, add a same-process unit test in `tests/test_cluster.py` that calls `cluster()` twice and asserts equal partitions, and rely on Phase 60's RED test `tests/test_e2e_integration.py::test_e2e_compose_override_ladder` (commit `333d2da`) as the cross-process gate. Cross-process determinism is what the bug requires; the unit test exists for cheap CI feedback, not as the primary witness.

## Common Pitfalls

### Pitfall 1: Missing the second Leiden call site
**What goes wrong:** Editing only `cluster()`'s call into `_partition` and missing `_split_community()`'s call into `_partition` (cluster.py:114).
**Why it happens:** `_split_community` invokes `_partition` for oversized communities; if `_partition` itself is not the edit site, the second call inherits the unseeded behavior.
**How to avoid:** The fix is **inside** `_partition` (line 38), so both call sites get the seed automatically. Verify by `grep -n "leiden(" graphify/cluster.py` post-edit ÔÇö there must be exactly one occurrence and it must include `random_seed=42`.
**Warning signs:** Test output where `cluster(G)` is stable on small graphs but flaky on graphs that trigger the >25% community split.

### Pitfall 2: Same-process trivial determinism
**What goes wrong:** Writing a unit test that calls `_partition()` twice in the same Python process, getting identical results even without the fix, and concluding the bug is fixed when it is not.
**Why it happens:** graspologic uses a process-scoped RNG; consecutive calls in the same process can return the same partition without an explicit seed if the RNG state happens to align.
**How to avoid:** The same-process unit test catches *regressions* (a future code change that re-introduces non-determinism within a single process). It is not the primary witness. The cross-process witness is Phase 60's E2E test, which forks a fresh `python -m graphify` subprocess for preview and another for apply. Both must be green.
**Warning signs:** Unit test passes locally but `pytest tests/test_e2e_integration.py::test_e2e_compose_override_ladder` still fails.

### Pitfall 3: Treating `random_seed=0` as "no seed"
**What goes wrong:** Using `random_seed=0` and assuming it means "unseeded" because of analogy to other libraries.
**Why it happens:** graspologic's signature is `random_seed: Optional[int] = None`. `None` is unseeded; `0` is a valid deterministic seed (just a different one). CONTEXT.md picks `42` to mirror the Louvain fallback; do not substitute `0`.
**How to avoid:** Use exactly `random_seed=42` per CONTEXT.md.
**Warning signs:** Plan IDs change after the patch is applied (suggests the seed value is being overridden somewhere).

## Process-Determinism Mental Model

Why `random_seed=None` is non-deterministic across processes (concise context for the planner):

- graspologic's Leiden uses a graph-internal RNG seeded from the OS source (effectively `secrets`/`os.urandom`-derived) when `random_seed` is `None`.
- Each fresh Python process has independent RNG state, so two `python -m graphify update-vault` invocations sample different seeds.
- The Leiden algorithm is deterministic given a fixed seed and fixed graph: same seed + same edge list Ôçĺ same partition labels in the same order.
- Graphify already canonicalizes downstream of the partition (sort by size, sort node lists in `cluster.py:103`), so the *only* unstable input to that canonicalization is the raw partition. Seeding it removes the entire non-determinism axis.

In-process determinism is "leaky" without a seed ÔÇö it can appear stable on small inputs and flake on larger ones ÔÇö which is exactly why CONTEXT.md selected the cross-process E2E gate as primary.
