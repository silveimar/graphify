# Phase 60.1 Context — update-vault apply determinism fix

<domain>
Make `graphify update-vault --apply --plan-id <id>` succeed on first run by eliminating non-determinism in the preview pipeline. Two consecutive `update-vault` preview runs on the same input + vault profile must produce the same `plan_id`. Root cause is `graspologic.partition.leiden()` invoked without `random_seed` in `graphify/cluster.py:_partition` (line ~36), so community partitions vary across separate Python processes. Phase 60's RED test `tests/test_e2e_integration.py::test_e2e_compose_override_ladder` (commit `333d2da`) is the regression gate.
</domain>

<code_context>
- **Bug site:** `graphify/cluster.py:30-42`. `_partition()` calls `from graspologic.partition import leiden; result = leiden(G)`. The `leiden` signature accepts `random_seed: Optional[int] = None` (verified via `inspect.signature`). Without a seed, partitions differ across separate process invocations.
- **Inconsistency:** `cluster.py:48` passes `seed=42` to the Louvain fallback. The Leiden path is the outlier.
- **Downstream:** `cluster()` re-indexes communities by size descending and sorts node lists (`cluster.py:103`), so community-id renumbering is already canonical *given a fixed partition* — but the input partition itself is non-deterministic, which the seed fixes.
- **plan_id flow:** `migration.py:162` `preview["plan_id"] = compute_migration_plan_id(preview)` over `_digest_payload` which sorts actions/paths/legacy_mappings. Action paths embed community IDs derived from cluster output, so any partition shift propagates into plan_id.
- **E2E witness:** memory 4736–4739 confirmed the RED test fails with `validate_plan_matches_request` raising "stale or mismatched migration plan" because preview and apply runs (separate subprocesses) produce different plan_ids.
- **Reusable test pattern:** `tests/test_cluster.py` already exists with NetworkX-graph fixtures; new determinism case slots in there.
</code_context>

<decisions>

### Fix strategy & depth — Seed Leiden only
Pass `random_seed=42` to `graspologic.partition.leiden()` in `graphify/cluster.py:_partition`. Surgical, matches the Louvain fallback's existing `seed=42`. Do **not** add label canonicalization (sorted node-set hashing) in this phase — the seed alone meets all four success criteria, and label canonicalization is a separate concern (graspologic version upgrades) that can be addressed if/when it becomes a real problem.

### Determinism guarantee scope — Cross-process subprocess
The guarantee is: two `python -m graphify update-vault` subprocess invocations on the same input corpus + vault profile produce identical `plan_id` values. Phase 60's E2E test (commit `333d2da`) is the cross-process witness. Same-process determinism is implied; no separate same-process-only standard.

### Regression test layer — Fast unit test + existing E2E
- **New:** add a unit test in `tests/test_cluster.py` that calls `cluster()` twice on the same `nx.Graph` fixture and asserts the two `dict[int, list[str]]` outputs are equal. Catches regressions in <1s on every CI run.
- **Existing:** Phase 60's `tests/test_e2e_integration.py::test_e2e_compose_override_ladder` (commit `333d2da`) must turn green **without modification**. This proves the cross-process subprocess scenario.
- No additional subprocess-level unit test — the E2E already covers it.

### Seed value & exposure — Hard-code `random_seed=42`
Pass `random_seed=42` inline at the `leiden(G, random_seed=42)` call site in `_partition()`. No module-level constant, no kwarg on `cluster()`, no env var. Matches Louvain fallback's `seed=42`. Zero new API surface; zero new config to document.

### Folded todos
None — no pending todos matched Phase 60.1 scope.

</decisions>
