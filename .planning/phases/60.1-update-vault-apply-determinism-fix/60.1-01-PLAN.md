<objective>
Fix the update-vault apply determinism bug at its root cause by seeding the Leiden community-detection call. Add a same-process unit test that fails on unseeded Leiden and passes after the one-line fix, following strict RED -> GREEN -> REFACTOR.

Purpose: Without `random_seed`, `graspologic.partition.leiden(G)` produces non-deterministic partitions across separate Python processes. Community IDs leak into action paths inside `migration._digest_payload`, causing `compute_migration_plan_id` to return different `plan_id` strings for two preview runs on the same input. This breaks `graphify update-vault --apply --plan-id <id>` on the first attempt and is the root cause flagged by Phase 60's E2E gate.

Output:
- A new failing-then-passing unit test `test_cluster_is_deterministic_across_runs` in `tests/test_cluster.py`.
- A one-line change to `graphify/cluster.py:38`: `leiden(G)` becomes `leiden(G, random_seed=42)`.
- Two atomic commits (RED, GREEN). REFACTOR step is a no-op (no cleanup needed for a one-line change) and is explicitly skipped.
</objective>
