## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Added deterministic sort tiebreaker in `cluster()`**

- **Found during:** Task 2 (GREEN) verification.
- **Issue:** After applying the planned one-line `random_seed=42` fix, the new test still failed ~40% of the time (8/20). Root cause investigation showed that while `_partition()` returns a deterministic `{node: cid}` mapping across processes, the dict's *insertion order* varies (graspologic's leiden uses an unordered internal structure that the seed does not stabilize). Downstream, `cluster()` builds `raw[cid] = [nodes...]` from that iteration order, then sorts by `len` descending. With Python's stable sort, equal-sized communities retain their input order — which varies — producing different community-ID assignments across runs.
- **Why this contradicts plan scope:** The plan stated the fix was "exactly one line at `graphify/cluster.py:38`" and forbade touching surrounding code. However, the locked truth `cluster(G1) == cluster(G2)` is **not satisfied** by seeding alone. APPLY-DET-01 cannot be closed without this second line. Per Rule 1 (auto-fix bugs that affect correctness/contract) the minimal additional change was applied.
- **Fix:** `graphify/cluster.py:103` — `final_communities.sort(key=len, reverse=True)` → `final_communities.sort(key=lambda c: (-len(c), sorted(c)))`. One line. No new imports, constants, kwargs, env vars, or refactors. Preserves the original ordering semantics (size-desc) and adds a deterministic tiebreaker (lexicographic on sorted node list) only when sizes tie.
- **Files modified:** `graphify/cluster.py` (one additional line beyond the planned one).
- **Commit:** `e110ead` (combined with the planned seed fix in the GREEN commit, since both are required to make the RED test pass).
